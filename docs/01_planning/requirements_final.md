## 1. 문서 개요

### 목적
TripCraft에 **실제 구현된** 기능을 사용자 관점으로 정의한다. 산출물(요구사항 명세·유스케이스·발표) 작성과
기능 검증의 기준 문서로 사용한다. 개발 전 계획(v0.1) 대비 실시간 공동편집·공유링크·커스텀 장소·
카카오 로그인·멀티모달 이동수단·AI 챗봇이 추가 구현되어 이를 모두 반영한다.

### 범위
국내 여행 계획: **관광지 탐색 → 후보군(보관함) 등록 → 드래그 앤 드롭 일정 확정 → 이동 시간 자동 계산 →
실시간 공동 편집·공유 → 커뮤니티 공유**. 관광지 데이터는 한국관광공사 TourAPI 기반.

### 용어
| 용어 | 의미 |
|------|------|
| 보관함(후보군) | 일정에 배치하기 전 모아두는 장소 목록 (`trip_candidate`) |
| 블록 | 타임라인(날짜·시간)에 확정 배치된 장소 (`trip_block`) |
| 커스텀 장소 | TourAPI에 없는 사용자 직접 입력/Kakao 검색 장소 |
| 협업자 | 일정에 초대된 EDITOR/VIEWER (`trip_collaborator`) |
| 공유 링크 | 토큰 기반 비로그인 조회/편집 접근 (`share_token`) |

## 2. Actor 정의

| Actor | 설명 | 인증 |
|-------|------|------|
| **비로그인 사용자(Guest)** | 랜딩·소개·관광지 탐색·커뮤니티 열람·공유 링크 일정 조회 | 불필요 |
| **회원(USER)** | 일정 작성·즐겨찾기·내 장소·커뮤니티 작성·AI 챗봇 등 전체 기능 | JWT |
| **일정 협업자 — EDITOR** | 초대받은 일정을 실시간 공동 편집 | JWT + 초대 |
| **일정 협업자 — VIEWER** | 초대받은 일정을 조회만 | JWT + 초대 |
| **관리자(ADMIN)** | 공지 관리, 관광지 데이터 동기화 | JWT + ROLE_ADMIN |
| **외부 시스템** | TourAPI · ODsay · T Map · Kakao(로그인·Local) · Naver Maps · gms(Spring AI) | API 키 |

## 3. 핵심 사용자 플로우

1. **온보딩**: 랜딩 → 회원가입/로그인(이메일 또는 카카오) → 작업실(`/plan`) 진입.
2. **계획 수립**: 관광지 탐색(지역·카테고리·키워드) → 보관함 담기(즐겨찾기/커스텀 포함) →
   날짜 탭에 드래그로 블록 배치 → 이동 시간 자동 계산 → 이동수단(대중교통/자동차/도보) 선택.
3. **협업·공유**: 협업자 초대(실시간 커서·블록 동기화) 또는 공유 링크 발급(조회/편집) → 타인의 공유 일정 복제.
4. **공유 게시**: 완성 일정을 커뮤니티에 게시 → 좋아요·댓글·북마크 → 마이페이지에서 관리.
5. **탐색 보조**: 관광지 상세에서 AI 챗봇으로 질의응답·주변 추천.

## 5. 비기능 요구사항

| 분류 | 요구사항 |
|------|----------|
| **보안** | 권한 검증은 서버에서 수행. JWT 쿠키 인증, 비밀번호 BCrypt. MyBatis `#{}` 바인딩만(SQL Injection 방지). WebSocket 채널 권한 검증(구독=조회권한, 발행=편집권한) |
| **성능** | 이동 시간·노선 폴리라인 캐싱, 프로필 이미지 N+1 제거, 관광지 DB 적재 후 조회(외부 API 호출 최소화) |
| **외부 API** | TourAPI 증분 동기화, ODsay/T Map 응답 모드별 캐시, 호출량 제한 대응 |
| **사용성** | 드래그 앤 드롭 일정 편집, 반응형 레이아웃, 실시간 협업 시 낙관적 락으로 끊김 없는 편집 |
| **호환성** | 최신 브라우저(Chrome 등), LAN 접속 허용(vite host) |
| **데이터 정합성** | 탈퇴 시 콘텐츠 보존 정책(작성자 SET NULL), 후보-블록 RESTRICT, 좌표 UNIQUE 중복 방지 |

## 6. 제약 사항 및 가정

- 국내 여행 한정(TourAPI·ODsay 국내 데이터).
- 외부 API 키·할당량에 의존(TourAPI·ODsay·T Map·Kakao·gms).
- 관광지 데이터는 배치 동기화 시점 기준(실시간 아님).
- 개발 인원 2인, 캡스톤 기간 내 구현.

## 7. Actor × Use-Case 카탈로그 (유스케이스 다이어그램 원본)

> 후속 유스케이스 **다이어그램** 작성 시 그대로 사용. (다이어그램 자체는 별도 이미지 산출)

| Actor | Use Case |
|-------|----------|
| **Guest** | 관광지 탐색, 관광지 상세 보기, 커뮤니티 열람, 공유 링크로 일정 조회, 회원가입/로그인 |
| **USER** | (Guest 전체) + 즐겨찾기, 내 장소 관리, 일정 생성·편집(보관함·블록·이동시간), 이동수단 선택, 협업자 초대, 공유 링크 발급, 공유 일정 복제, 게시글·댓글·좋아요·북마크, AI 챗봇 질의, 프로필 관리, 회원 탈퇴 |
| **EDITOR** | 초대된 일정 실시간 공동 편집(블록·이동수단), 프레즌스 공유 |
| **VIEWER** | 초대된 일정 실시간 조회 |
| **ADMIN** | (USER 전체) + 공지 작성·수정·삭제, 관광지 데이터 동기화 |
| **외부 시스템** | TourAPI(관광지), ODsay(대중교통), T Map(자동차·도보), Kakao(로그인·장소검색), Naver Maps(지도), gms/Spring AI(챗봇) |
