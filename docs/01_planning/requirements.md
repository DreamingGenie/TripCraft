## 1. 문서 개요

### 목적

본 문서는 TripCraft 프로젝트의 기능 요구사항을 사용자 스토리(User Story)와 상세 명세 형식으로 정의한다. 개발팀(백엔드·프론트엔드), 설계(ERD·API 명세), 테스트(QA) 작업의 기준 문서로 활용한다.

### 표기 규칙

- **[필수]**: 해당 기능이 없으면 MVP 완성 불가. 일정 내 반드시 구현.
- **[권장]**: MVP 범위이나 일정 압박 시 범위 축소 가능. 별도 표기.
- **[Phase 2]**: MVP 범위 외. 안정화 후 다음 스프린트에서 구현.
- 수용 기준(Acceptance Criteria, AC)은 QA·테스트 기준으로 활용한다.

## 2. 사용자 역할 (Actor) 정의

| 역할 | 설명 | 주요 권한 |
|------|------|---------|
| **비회원 (Guest)** | 로그인하지 않은 방문자 | 관광지 조회, 커뮤니티 열람, 공지사항 열람 |
| **회원 (Member)** | 로그인한 일반 사용자 | 비회원 권한 + 일정 생성·편집·저장, 즐겨찾기, 게시글 작성 |
| **관리자 (Admin)** | 서비스 운영자 | 회원 권한 + 공지사항 CRUD, 게시글 관리 |

## 3. 핵심 사용자 플로우

서비스의 핵심 시나리오를 플로우 형태로 기술한다. 세부 화면 명세는 Figma 명세서(`docs/02_design/wireframe_spec.md`) 참조.

```
Step 1. 일정 생성
  ─ 회원이 GNB "내 일정" 탭 클릭 또는 "+ 새 일정 만들기" 선택
  ─ 일정 생성 모달에서 여행 제목·출발일·귀환일·인원 입력
  ─ "만들기" 클릭 → 일정 생성, 화면 B(내 일정)로 이동

Step 2. 후보군 등록 (화면 A — 관광지 탐색)
  ─ 지역 / 카테고리 필터로 관광지 탐색
  ─ 원하는 장소 카드에서 "일정에 추가 +" 클릭
  ─ 선택 장소가 활성 일정의 후보군에 등록됨 (도시별 자동 분류)
  ─ 해당 도시 첫 등록 시: 즐겨찾기 동일 도시 장소가 후보군에 자동 추가 + 토스트 알림

Step 3. 일정 확정 (화면 B — 내 일정)
  ─ 좌측 후보군 사이드바(도시 그룹별 CandidateCard) 확인
  ─ CandidateCard를 우측 타임라인 DropZone으로 드래그 앤 드롭
  ─ PlaceBlock 생성 → ODsay API로 이전·이후 장소와의 이동 시간 자동 계산·삽입
  ─ 블록 재배치 / 삭제 / 시간 수정 → 이동 시간 재계산

Step 4. 일정 공유 (화면 C — 커뮤니티)
  ─ 완성된 일정을 커뮤니티 게시판에 공유
  ─ 다른 사용자의 공유 일정 열람 및 참고
```

### F12 — 회원 관리

> 가장 먼저 구현해야 하는 기반 기능. 이후 모든 개인화 기능의 전제 조건.

#### 사용자 스토리

| ID | 역할 | 행동 | 목적 |
|----|------|------|------|
| US-12-01 | 방문자 | 이메일·닉네임·비밀번호로 회원가입을 한다 | 서비스의 개인화 기능을 이용하기 위해 |
| US-12-02 | 방문자 | 이메일·비밀번호로 로그인한다 | 내 일정과 즐겨찾기에 접근하기 위해 |
| US-12-03 | 회원 | 로그아웃한다 | 공용 기기에서 내 계정을 보호하기 위해 |
| US-12-04 | 회원 | 닉네임·비밀번호를 수정한다 | 최신 정보로 계정을 유지하기 위해 |
| US-12-05 | 회원 | 계정을 탈퇴한다 | 서비스에 저장된 내 데이터를 삭제하기 위해 |

#### 기능 명세

**회원가입 [필수]**
- 입력 항목: 이메일, 닉네임, 비밀번호, 비밀번호 확인
- 이메일: 유효한 형식(`xxx@xxx.xxx`), 중복 불가
- 닉네임: 2~20자, 특수문자 제한
- 비밀번호: 8자 이상, 영문·숫자 포함 필수
- 비밀번호는 BCrypt로 해시하여 저장
- 회원가입 후 자동 로그인 처리

**로그인 / 로그아웃 [필수]**
- 로그인 성공 시 Access Token (30분) + Refresh Token (7일) 발급
- Access Token은 HTTP 헤더(`Authorization: Bearer`)로 전달
- Refresh Token은 HttpOnly 쿠키로 관리
- 로그아웃 시 Refresh Token 무효화

**회원 정보 수정 [필수]**
- 수정 가능 항목: 닉네임, 비밀번호
- 비밀번호 변경 시 현재 비밀번호 재확인

**회원 탈퇴 [필수]**
- 비밀번호 재확인 후 탈퇴 처리
- 탈퇴 시 일정·즐겨찾기·좋아요는 삭제. 작성한 게시글·공지는 보존 ("탈퇴한 사용자"로 표시)

#### 수용 기준 (Acceptance Criteria)

- AC-12-01: 이미 사용 중인 이메일로 가입 시 "이미 사용 중인 이메일입니다." 오류 메시지 노출
- AC-12-02: 비밀번호 확인 불일치 시 가입 버튼 비활성화
- AC-12-03: 잘못된 이메일 또는 비밀번호 입력 시 로그인 실패 메시지 노출, 계정 정보 노출 없음
- AC-12-04: Access Token 만료 시 Refresh Token으로 자동 재발급됨
- AC-12-05: 로그인하지 않은 상태에서 보호된 페이지 접근 시 로그인 페이지로 리다이렉트
- AC-12-06: 탈퇴 후 해당 계정으로 로그인 불가

### F01 · F02 · F03 — 관광지 DB 구축 및 조회

#### 사용자 스토리

| ID | 역할 | 행동 | 목적 |
|----|------|------|------|
| US-01-01 | 시스템 | 한국관광공사 API에서 관광지 데이터를 수집하여 DB에 저장한다 | 사용자가 외부 사이트 이탈 없이 관광지를 탐색할 수 있도록 |
| US-02-01 | 비회원/회원 | 시도·시군구 단위로 관광지를 필터링하여 조회한다 | 여행 목적지의 장소만 모아서 보기 위해 |
| US-03-01 | 비회원/회원 | 관광지·음식점·숙박·문화시설 등 카테고리별로 조회한다 | 원하는 종류의 장소를 빠르게 찾기 위해 |
| US-03-02 | 비회원/회원 | 관광지명으로 검색한다 | 특정 장소를 이름으로 직접 찾기 위해 |
| US-03-03 | 비회원/회원 | 관광지 카드를 클릭해 상세 정보를 확인한다 | 방문 전 전화번호·주소·소개 등을 확인하기 위해 |

#### 기능 명세

**관광 DB 구축 [필수]**
- 한국관광공사 국문 관광정보 서비스 API(`TourAPI 4.0`) 활용
- 수집 대상: 관광지(12), 문화시설(14), 레포츠(28), 숙박(32), 쇼핑(38), 음식점(39)
- 시도·시군구 코드 기준 전체 데이터 일괄 수집 후 DB 저장 (초기 1회 배치)
- API 직접 호출 최소화 — 수집 후 자체 DB에서 서비스
- 결측 데이터(이미지 없음, 주소 없음 등) 허용, 별도 처리 불필요

**지역별·카테고리별 조회 [필수]**
- 지역 필터: 시도(다중 선택 가능) → 시군구로 세분화
- 카테고리 필터: 콘텐츠 타입 코드 기준 (다중 선택 가능)
- 검색: 관광지명 키워드 검색 (부분 일치)
- 목록: 카드 2열 그리드, 페이지네이션(또는 무한 스크롤)
- 지도 연동: 목록의 장소가 네이버맵에 마커로 표시됨. 핀 클릭 시 InfoWindow, 상세 슬라이드 패널 연동

**관광지 상세 [권장]**
- 장소명, 카테고리, 주소, 전화번호, 대표 이미지, 소개(overview) 노출
- 지도에서 해당 마커 선택 시 목록에서 해당 카드 하이라이트

#### 수용 기준

- AC-01-01: DB에 관광지 데이터가 적재되어 있으며 지역·카테고리 필터 결과가 정확히 반환됨
- AC-02-01: 시도 선택 후 해당 시도 소속 관광지만 노출됨
- AC-03-01: 카테고리 필터 선택 시 해당 콘텐츠 타입 코드에 매핑된 결과만 노출됨
- AC-03-02: 검색어 입력 시 장소명에 해당 키워드가 포함된 결과만 노출됨
- AC-03-03: 지역·카테고리 필터 미선택 시 전체 목록 반환됨

### F04 — 즐겨찾기

#### 사용자 스토리

| ID | 역할 | 행동 | 목적 |
|----|------|------|------|
| US-04-01 | 회원 | 관광지 카드의 별 아이콘을 클릭해 즐겨찾기를 추가/해제한다 | 나중에 다시 찾기 쉽도록 관심 장소를 저장하기 위해 |
| US-04-02 | 회원 | 필터 사이드바에서 내 즐겨찾기 목록을 확인한다 | 저장해 둔 장소를 한곳에서 모아 보기 위해 |
| US-04-03 | 회원 | 일정에 특정 도시가 처음 추가될 때 그 도시의 즐겨찾기 장소가 후보군에 자동 추가된다 | 즐겨찾기해 둔 장소를 빠뜨리지 않고 일정에 반영하기 위해 |

#### 기능 명세

**즐겨찾기 토글 [필수]**
- 로그인 상태에서만 가능. 비회원 클릭 시 로그인 유도 메시지 노출
- 동일 장소 중복 등록 불가 (DB UNIQUE 제약)
- 아이콘 상태: 미등록(`ti-star`, 기본) ↔ 등록됨(`ti-star-filled`, amber)

**즐겨찾기 목록 [필수]**
- 관광지 탐색 화면(A) 필터 사이드바 하단에 즐겨찾기 장소 목록 표시
- 클릭 시 해당 장소 카드로 포커스 이동 또는 상세 정보 표시

**도시 첫 추가 시 자동 연동 [필수]**
- 후보군에 특정 도시(시군구 단위)의 장소가 처음 추가될 때 트리거
- 즐겨찾기 중 동일 도시(시군구 코드 일치) 장소를 후보군에 일괄 추가
- 토스트 메시지: "[도시명]의 즐겨찾기 N개가 후보군에 추가됐어요" (3초 자동 소멸)
- 이미 후보군에 있는 장소는 중복 추가하지 않음

#### 수용 기준

- AC-04-01: 즐겨찾기 추가 후 사이드바 즐겨찾기 목록에 즉시 반영됨
- AC-04-02: 즐겨찾기 해제 후 목록에서 제거됨
- AC-04-03: 비회원이 즐겨찾기 클릭 시 "로그인이 필요합니다" 메시지 노출
- AC-04-04: 서울 즐겨찾기 3개 등록 후 서울 장소 최초 후보군 추가 시 3개가 자동으로 후보군에 추가되고 토스트 표시됨
- AC-04-05: 자동 추가 시 이미 후보군에 있는 장소는 중복 추가되지 않음

### F05 — 일정 생성

#### 사용자 스토리

| ID | 역할 | 행동 | 목적 |
|----|------|------|------|
| US-05-01 | 회원 | "+ 새 일정 만들기"를 클릭해 일정 생성 모달을 연다 | 새로운 여행 계획을 시작하기 위해 |
| US-05-02 | 회원 | 여행 제목·출발일·귀환일·인원을 입력하고 일정을 만든다 | 여행의 기본 정보를 설정하기 위해 |
| US-05-03 | 회원 | 내 일정 목록에서 기존 일정을 선택해 편집을 재개한다 | 이전에 만든 일정을 이어서 작업하기 위해 |

#### 기능 명세

**일정 생성 모달 [필수]**
- 트리거: GNB "내 일정" 탭 최초 진입 시 또는 "+ 새 일정 만들기" 클릭
- 필수 입력: 여행 제목(최대 30자), 출발일, 귀환일
- 선택 입력: 여행 인원(기본값 1)
- 날짜 선택 시 "총 N박 N일" 자동 계산 표시
- 필수 항목 미입력 시 "만들기" 버튼 disabled
- 유효성: 출발일 ≤ 귀환일

**일정 목록 [필수]**
- 회원별 일정 목록을 최신 수정순으로 정렬 표시
- 활성 일정 전환: 일정명 드롭다운으로 다른 일정으로 전환 가능

#### 수용 기준

- AC-05-01: 여행 제목 미입력 시 만들기 버튼 비활성화
- AC-05-02: 귀환일을 출발일보다 빠르게 선택 시 오류 표시
- AC-05-03: 일정 생성 완료 후 화면 B(내 일정)로 자동 이동되며 새 일정이 활성화됨
- AC-05-04: 기존 일정 선택 시 해당 일정의 후보군과 타임라인이 로드됨

### F06 — 후보군 등록

#### 사용자 스토리

| ID | 역할 | 행동 | 목적 |
|----|------|------|------|
| US-06-01 | 회원 | 관광지 카드에서 "일정에 추가 +"를 클릭해 후보군에 등록한다 | 가고 싶은 장소를 일단 모아두기 위해 |
| US-06-02 | 회원 | 등록된 후보군이 도시(시군구)별로 자동 묶여 표시되는 것을 확인한다 | 어느 도시에 어떤 장소를 방문할지 한눈에 파악하기 위해 |
| US-06-03 | 회원 | 활성 일정이 없는 상태에서 후보군 추가를 시도하면 일정 선택/생성을 안내받는다 | 혼란 없이 올바른 플로우로 유도되기 위해 |

#### 기능 명세

**후보군 추가 [필수]**
- 활성 일정이 있을 때: 클릭 즉시 후보군 등록 + 카드 상태 `Candidate`로 전환
- 활성 일정이 없을 때: 일정 선택 드롭다운 또는 "새 일정 만들기" 유도
- 동일 장소 중복 추가 불가 (이미 추가된 경우 버튼 "추가됨" 비활성)

**도시별 자동 분류 [필수]**
- 후보군은 장소의 시군구 코드 기준으로 도시 그룹으로 묶임
- 도시 그룹은 CityGroup 컴포넌트로 표시 (도시명 + 장소 수 + 접기/펼치기)
- 장소 추가 순서대로 그룹 내 정렬

**후보군 삭제 [권장]**
- CandidateCard 우측 삭제 아이콘으로 후보군에서 제거
- 이미 타임라인에 배치된(Placed) 장소는 후보군 삭제 시 타임라인 블록도 함께 제거 여부 확인 모달 표시

#### 수용 기준

- AC-06-01: 후보군 추가 후 AttractionCard가 `Candidate` 상태로 변경되고 "추가됨" 버튼이 비활성화됨
- AC-06-02: 내 일정 탭의 사이드바에 추가한 장소가 도시별로 그룹화되어 표시됨
- AC-06-03: 같은 장소를 두 번 추가 시도 시 중복 추가되지 않음

#### 사용자 스토리

#### 기능 명세

**날짜 탭 [필수]**
- 일정 생성 시 날짜 범위 기준으로 Day 1 ~ Day N 탭 자동 생성
- 활성 탭: 보라 배경 + 흰색 텍스트

**드래그 앤 드롭 [필수]**
- 후보군 CandidateCard → 타임라인 DropZone으로 드래그
- 드래그 오버 시 DropZone 활성 상태 표시 (보라색 점선 테두리 + 배경색)

- **중복 방문 허용**: 이미 배치된 CandidateCard도 재드래그하여 같은 장소를 여러 번 배치 가능
- PlaceBlock 재배치: 다른 날짜 또는 다른 시간대 슬롯으로 드래그 앤 드롭으로 이동

**PlaceBlock 삭제 [필수]**
- PlaceBlock hover 시 우측 상단에 X 아이콘 노출, 클릭 시 삭제
- PlaceBlock을 좌측 후보군 사이드바로 드래그하면 사이드바가 빨간 삭제 존으로 변하고 드롭 시 삭제

**체류 시간 조정 [필수]**
- PlaceBlock 하단에 리사이즈 핸들 노출 (hover 시 표시)
- 핸들을 아래로 드래그 → 30분 단위로 스냅하며 체류 시간(duration_minutes) 조정
- 마우스 업 시 서버에 자동 저장

**PlaceBlock 상세 편집 [권장]**
- 클릭 시 메모 입력 가능
- 시작 시간 직접 수정 가능 (이후 블록 시간 연쇄 이동 여부 선택)

#### 수용 기준

- AC-07-01: CandidateCard 드래그 시작 시 해당 카드의 ghost 이미지가 커서를 따라 이동함
- AC-07-02: 드롭 완료 후 타임라인에 PlaceBlock이 생성되고 CandidateCard가 반투명·취소선 상태로 변경됨
- AC-07-03: PlaceBlock X 클릭 후 블록 삭제 + CandidateCard가 정상 상태로 복원됨

- AC-07-06: PlaceBlock을 사이드바로 드래그 시 사이드바가 삭제 존으로 표시되고 드롭 시 블록 삭제됨
- AC-07-07: 체류 시간 핸들 드래그 시 30분 단위로 스냅되며 마우스 업 시 서버에 저장됨

### F08 — 이동 시간 자동 계산

#### 사용자 스토리

| ID | 역할 | 행동 | 목적 |
|----|------|------|------|
| US-08-01 | 회원 | 타임라인에 장소를 배치하면 이전·이후 장소와의 이동 시간이 자동으로 표시된다 | 별도로 이동 시간을 검색하지 않아도 현실적인 일정을 짜기 위해 |
| US-08-02 | 회원 | 이동 수단(대중교통 / 도보 / 자동차)과 소요 시간을 TransitPill에서 확인한다 | 이동 방법과 시간을 한눈에 파악하기 위해 |
| US-08-03 | 회원 | PlaceBlock별로 선호 이동수단(대중교통·지하철·버스·자동차)을 선택한다 | 구간마다 다른 이동수단을 유연하게 사용하기 위해 |

#### 기능 명세

**이동 시간 자동 계산 및 삽입 [필수]**
- 트리거: PlaceBlock 배치 / 재배치 / 삭제 시
- 대중교통(ODsay API) 또는 자동차(별도 API) 중 PlaceBlock 선호 이동수단 기준으로 호출
- 계산 결과를 PlaceBlock 사이에 TransitPill로 자동 삽입
- TransitPill 표시: 이동 수단 아이콘(버스/지하철/도보/자동차) + 소요 시간(분)
- 복수 환승 경로는 대표 수단만 표시 (최단 시간 기준)

**구간별 이동수단 선택 [필수]**
- PlaceBlock마다 선호 이동수단 설정 가능: 대중교통 전체 / 지하철 / 버스 / 자동차
- 기본값: 대중교통 전체

**API 호출 최적화 [필수]**
- 캐시 키: (출발지, 도착지, 출발 시간대, 이동수단) 조합으로 저장
- 캐시 히트 시 API 호출 없이 즉시 반환
- 시간대별 캐시 정밀도는 관리자 설정(1~5단계)으로 조정 가능
- API 오류 또는 타임아웃(5초 초과) 시: "이동 시간을 계산할 수 없습니다" 표시, 일정 저장은 정상 가능

**위치 기반 한계 처리 [필수]**
- ODsay 무료 플랜의 도보 경로는 직선거리 기반 예상 시간으로 표시
- 대중교통 경로가 없는 경우(도서·산간 지역 등): "경로 없음" 표시

#### 수용 기준

- AC-08-01: PlaceBlock 배치 후 1~2초 이내 TransitPill이 자동 삽입됨
- AC-08-02: 동일 구간 재배치 시 API 재호출 없이 캐시된 시간이 즉시 표시됨
- AC-08-03: API 응답 실패 시 "이동 시간 계산 불가" 표시 + 나머지 일정은 정상 동작
- AC-08-04: PlaceBlock 삭제 시 인접 TransitPill이 제거되고 새 인접 구간의 이동 시간이 자동 계산됨
- AC-08-05: PlaceBlock별 이동수단 변경 시 해당 구간 TransitPill이 선택 수단 기준으로 재계산됨

## 6. 비기능 요구사항

### 6-1. 성능

| 초기 페이지 로드 | 3초 이하 |

### 6-2. 보안

| 항목 | 요구사항 |
|------|---------|
| 인증 | JWT (Access 30분 + Refresh 7일), HttpOnly 쿠키 |
| 비밀번호 | BCrypt 해시 저장, 평문 저장 절대 금지 |
| SQL Injection | MyBatis `#{}` 바인딩 필수 |
| 인가 | 모든 보호 API에 서버 측 소유권 검증 |
| API 키 | 환경 변수 관리, 소스코드에 하드코딩 금지 |
| CORS | 허용 Origin 명시적 설정 |

### 6-3. 사용성

| 항목 | 요구사항 |
|------|---------|
| 반응형 | 데스크톱(1280px 이상) 우선. 태블릿(768px) 기본 지원 |
| 로딩 상태 | API 호출 중 스피너 또는 스켈레톤 UI 표시 |
| 오류 상태 | 사용자 친화적 오류 메시지 (기술적 스택 트레이스 노출 금지) |
| 빈 상태 | 데이터 없음 상태(Empty State) 화면 구성 |

### 6-4. 유지보수성

| 항목 | 요구사항 |
|------|---------|
| 코드 컨벤션 | `docs/03_dev/conventions.md` 준수 |
| API 문서 | Swagger UI (`/swagger-ui.html`) 제공 |
