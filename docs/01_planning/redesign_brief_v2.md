# TripCraft — 화면 전면 재기획 브리프 (v2, 디자이너용)

> **목적**: 디자이너(사람/AI)가 **사용성 최우선의 이상적 UI/UX**를 백지에서 기획하도록 제품의 본질을
> 한 장에 정리한다.
>
> 🟢 **이 기획의 원칙 — 완전 제약 없음**
> - **사용성이 최우선.** 더 쉽고·빠르고·명확한 경험이면 무엇이든 제안 가능.
> - **기존 구현/동작 보존은 제약이 아니다.** 화면 구성·내비게이션·인터랙션·플로우·기능을 **백지에서 자유롭게** 다시 설계해도 된다.
> - 아래 §5~§6의 기술/데이터 항목은 "지금 이런 게 있다"는 **참고 맥락**일 뿐, 지켜야 할 규칙이 아니다.
> - 구현 가능성은 **이후 별도 판단**. 지금은 이상적 UX만 그린다.
>
> 기능 상세는 재서술하지 않고 [`requirements.md`](./requirements.md) 참조.
> 비주얼 토큰/원칙은 [`design_system.md`](../03_dev/design_system.md) 참조.
> 1차 리디자인 결과(현 상태)는 [`ui_redesign_v1.md`](../02_design/ui_redesign_v1.md) — 이번은 그 위 2차 재기획.

## 1. 제품 정의 · 목적
**TripCraft** — 지도 기반 국내 여행 일정 플래너.
한 문장: *"관광지를 지도에서 탐색 → 후보로 담기 → 드래그로 시간표에 확정 → 이동시간 자동 계산 → 커뮤니티 공유"* 까지 한 흐름으로 끝내는 도구.

- **데이터 출처**: 한국관광공사 TourAPI(관광지), ODsay/TMap(대중교통·도보·자동차 이동시간), Naver Maps(지도).
- **성격**: 단순 조회 앱이 아니라 **"계획을 만드는 작업 도구(workspace)"**. 정보 밀도가 핵심 가치.

## 2. 사용자 (Actor)
| 역할 | 할 수 있는 것 |
|------|--------------|
| 비회원 | 관광지 조회, 커뮤니티·공지 열람 |
| 회원 | + 일정 생성·편집·저장, 후보군, 즐겨찾기, 게시글, AI 챗봇 |
| 관리자 | + 공지 CRUD, 게시글 관리 |

## 3. 핵심 플로우 (이 골격은 유지)
```
일정 생성 → [탐색] 관광지 후보 담기 → [내 일정] 드래그로 시간표 확정
         → 이동시간 자동 계산 → [커뮤니티] 공유
보조: [여행일지] 달력으로 내 일정 조망 · [마이페이지] 내 활동 관리 · [AI 챗봇] 주변 추천
```
일정 1개가 **탐색·일정·달력·공유를 관통하는 중심 객체**(= "현재 일정"). 화면 간 현재 일정은 동기화됨(`stores/activeTrip.js`).

## 4. 화면 인벤토리 + 역할(JTBD) + 핵심 요소

### A. 관광지 탐색 `/explore` — **workspace(고밀도)**
- **역할**: "어디 갈지 찾아서 일정에 담는다."
- **핵심 요소**: 풀스크린 지도(배경) · 좌측 플로팅 검색/필터/결과 패널 · 결과 카드(이미지·분류·지역) ·
  상세 슬라이드인 패널(히어로 이미지/갤러리/주소·전화·홈페이지/운영정보 테이블/소개/AI 챗봇) ·
  우하단 "내 일정" 장바구니(현재 일정 select + 담은 후보 + 드롭존).
- **데이터 현실**: 결과는 **지역(시도)>시군구>분류 3단**으로 그룹/카운트되고 그룹 단위 lazy-load + 무한스크롤.
  필터 칩(지역/시군구/분류). 분류 = 6종 콘텐츠타입(관광지·문화시설·레포츠·숙박·쇼핑·음식점).

### B. 내 일정 `/schedule` — **workspace(최고밀도, 시그니처)**
- **역할**: "담은 후보를 날짜·시간에 배치해 일정을 확정한다."
- **핵심 요소**: 상단 툴바(일정 select·공유·새 일정·지도보기) · 좌측 후보군 사이드바(도시>시군구>분류 트리, 드래그 핸들) ·
  중앙 **24시간 타임테이블**(요일 컬럼 × 시간축, 이벤트 블록 드래그/리사이즈/삭제) ·
  이벤트 사이 **이동 pill**(분/수단, 클릭 시 수단 변경 드롭다운) · 지도 패널(대중교통 구간·도보 폴리라인).
- **데이터 현실**: 블록 배치 시 ODsay로 앞뒤 이동시간 자동 계산·삽입, 재배치 시 재계산(비동기·로딩). 분류색 = A와 동일.

### C. 커뮤니티 `/community`, `/community/:id` — **content(저밀도)**
- **역할**: "완성 일정/여행기를 공유하고 남의 것을 본다."
- **핵심 요소**: 글 목록(작성자 아바타·제목·좋아요/댓글/조회) · 정렬·검색 · 공지 배너 ·
  글쓰기 모달(Tiptap 에디터 + 일정 연결 select) · 상세(본문·일정 카드·댓글·대댓글·좋아요·북마크).

### D. 여행일지 `/calendar` — **content**
- **역할**: "내 일정들을 달력에서 한눈에." 월 그리드 + 일정 띠(Google 캘린더식) + 날짜 상세 패널.

### E. 마이페이지 `/mypage/*` — **content**
- **역할**: "내 활동 관리." 탭: 내 정보 / 방문지도 / 내가 쓴 글 / 북마크 / 좋아요. (현재 가장 최신 톤 = 콘텐츠 화면 레퍼런스)

### F. 진입/주변 — Landing `/`, About, Auth `/auth`(+카카오), 404, Admin.

## 5. 현재 데이터·인터랙션 맥락 (참고용 — 제약 아님)
"지금 제품이 다루는 데이터·상호작용"이다. 지켜야 할 규칙이 아니라 **무엇을 다루는 제품인지** 이해하기 위한 배경.
더 나은 UX를 위해 구조·흐름·표현을 **자유롭게 바꿔도 된다**(괄호 안은 개선 여지 힌트).

1. **지도** — 관광지는 좌표가 있어 지도에 표시됨(Naver). (지도 비중·표현 자유롭게 재구성 가능)
2. **장소를 일정에 담고 배치하는 행위가 핵심** — 현재는 드래그 앤 드롭. (클릭 추가·자동 배치·추천 순서 등 더 쉬운 방식으로 바꿔도 됨)
3. **일정은 날짜·시간대를 가짐** — 현재 24h 타임테이블. (타임라인/리스트/카드 등 더 읽기 쉬운 표현 자유)
4. **관광지는 지역(시도>시군구)·분류(6종)로 나뉨** — 양이 많음. (3단 트리 고집할 필요 없음; 필터·검색·점진노출 등 자유)
5. **분류 6색은 좋은 통일 장치** — 지도·카드·일정·달력에서 같은 색이면 인지가 쉬움(`design_system.md` `--cat-*`). 권장이되 강제 아님.
6. **데이터 모양**: 관광지(이미지·주소·전화·운영정보·소개), 일정(장소+이동시간), 글(작성자·통계·일정연결), 댓글/좋아요/북마크. (표현 방식 자유)
7. 이동시간(대중교통/도보/자동차)은 자동 계산 가능한 정보다 — 어떻게 보여줄지 자유.

## 6. 현재 기술 스택 (참고 — 제약 아님)
- Vue 3 + Vite + Pinia + Vue Router. 외부: Naver Maps / TourAPI / ODsay·TMap(이동시간) / Tiptap(에디터).
- 인증: 쿠키 JWT. — *기획 단계에선 신경 쓰지 말 것. 필요하면 새 화면·기능·데이터도 제안 OK.*

## 7. 디자인 방향 (권장 출발점 — 더 나으면 제안 가능)
- **Warm Editorial Workspace**(`design_system.md`): 따뜻한 종이 베이스 + 퍼플 액센트, 위계는 타이포로·색은 절제,
  테두리보다 그림자, 이모지 금지(라인 SVG 1세트), 분류 6색 공통.
- 밀도 2단: 작업 화면(탐색·일정) = dense, 콘텐츠 화면(커뮤니티·달력·마이) = airy. 같은 토큰에서 파생.
- 이 톤이 최선이 아니라고 보면 **대안 방향을 제안해도 된다**(사용성·여행 도메인 적합성 근거와 함께).

## 8. 범위
- 이번 산출물은 **UI/UX 기획**(IA·플로우·와이어프레임·비주얼). 구현 방법은 다루지 않는다.
- **제외 없음** — 사용성에 도움이 되면 화면 합치기/쪼개기/신설, 새 기능·데이터, 플로우 변경 무엇이든 제안 가능.
- 단, "TripCraft = 지도 기반 국내 여행 일정 플래너(탐색→계획→공유)"라는 **제품 본질**은 유지.

## 9. 디자이너에게 바라는 산출물
1. **전체 IA/내비게이션** 재정리 (자유).
2. **사용자 플로우 재설계** — 더 적은 단계로 핵심 과업(탐색→담기→일정 확정→공유)을 끝내는 흐름.
3. **화면별 와이어프레임**(요소 배치·우선순위·상태: 빈/로딩/에러). 화면 합치기/쪼개기/신설 자유.
4. **비주얼 방향 적용 예시** 1~2 화면.
5. 각 결정에 **사용성 근거 1줄**(왜 이게 더 쉬운가).

**최적화할 사용성 기준**: 학습 용이성(처음 와도 다음 할 일이 보임) · 효율(클릭·단계 최소) ·
명확한 피드백(담김/계산 중/저장됨) · 실수 방지·복구 · 일관성 · 적절한 정보 밀도.
