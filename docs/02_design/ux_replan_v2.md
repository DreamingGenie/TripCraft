## 1. 설계 관점 / 핵심 UX 문제 진단

현재 구현을 "지켜야 할 것"이 아니라 "지금 사용자가 무엇 때문에 막히는가"의 증거로 읽었다. 코드(`ExploreView.vue` 1,503줄 / `ScheduleView.vue` 1,682줄)에서 반복적으로 드러나는 구조적 문제 3가지를 먼저 못 박는다.

### 문제 ① "찾는 화면"과 "짜는 화면"이 분리되어 같은 데이터를 두 번 학습해야 한다 (가장 큰 문제)
- 지금은 **탐색(`/explore`)에서 후보를 담고 → 일정(`/schedule`)으로 이동해 다시 배치**한다. 두 화면 모두 지도·패널·드래그를 쓰는 고밀도 workspace인데, **데이터가 서로 다른 모양**(탐색 = 지역>시군구>분류 결과 트리 / 일정 = 도시>시군구>분류 후보 트리 + 24h 타임테이블)으로 두 번 등장한다.
- 사용자는 "방금 담은 그 장소가 일정의 어디에 들어갔지?"를 **화면을 떠나야** 확인한다. 핵심 과업(찾기→담기→배치)이 **물리적으로 단절**돼 있다.
- **[근거: 학습성·일관성]** 같은 객체(장소)를 한 화면 안에서 보고-담고-배치하면 멘탈 모델이 하나로 합쳐진다. 화면 전환 = 컨텍스트 비용.

### 문제 ② "3단 트리"가 탐색·후보군 양쪽을 지배해 클릭 비용과 인지 부하가 크다
- 탐색 결과: **지역(시도) > 시군구 > 분류** 3단 그룹 + 그룹별 lazy-load.
- 후보군 사이드바: **도시 > 시군구 > 분류** 3단 아코디언(`city-header > sigungu-header > cat-header`, chevron 3겹).
- 장소 하나 보려면 펼치기를 최대 3번. 트리는 "전체 구조 파악"엔 좋지만 "내가 담은 7개를 빨리 본다"엔 과하다.
- **[근거: 효율·정보밀도]** 여행 후보는 보통 한 도시 10~20곳 규모. 이 규모엔 트리보다 **필터 + 플랫 리스트/지도 동기화**가 클릭당 정보량이 높다.

### 문제 ③ 24h 타임테이블의 **인터랙션이 드래그·리사이즈 한 경로에만 의존**해 실수를 부르고 진입을 막는다

> **★결정(2026-06-23): 24시간 격자 타임테이블은 유지한다.** 정밀 시간 제어와 이동시간 시각화는 이 제품의 시그니처이자 시연·채점 포인트라 보존한다. 진단의 초점은 "격자를 버리자"가 아니라 **"격자 위의 인터랙션만 개선하자"**로 좁힌다.

- 격자 자체는 강점이다: 요일 컬럼 × 24시간 축은 **하루 동선의 시간 밀도·과밀·빈 시간대를 한눈에** 보여주고, 블록 사이 이동 pill이 정밀한 시각 위치를 가질 수 있다. → **유지.**
- 문제는 **조작 경로가 "픽셀 드래그 + 핸들 리사이즈" 단 하나**라는 점이다. 시작시간을 직접 잡아야 하고, 30분 스냅 리사이즈를 손으로 끌어야 하며, **드래그를 못 하면(터치·키보드·정밀조준 실패) 일정을 한 칸도 못 채운다.**
- 삭제가 "사이드바(삭제 존)로 드래그"여서 **오발 제거 위험**이 있고, 한 번 빼면 어디로 갔는지 불명확하다.
- 이동 pill의 비동기 계산 상태(계산 중·실패)가 약하게 노출돼 "왜 안 뜨지?"가 생긴다.
- 빈 일정에서 "이제 뭘 해야 하지?"가 안 보인다(빈 격자만 있음).
- **[근거: 실수방지·효율·학습성]** 격자는 그대로 두되, **클릭 추가를 기본 경로로 추가**(드래그는 보너스), **빼기=보관함 복귀**(손실 0), 시작시간 자동 제안, 이동 pill 인라인 상태 노출. 드래그를 못 해도 100% 완수 가능하게.

### 부차 문제
- **현재 일정(activeTrip)이 toolbar select에 숨어 있다.** 전 화면을 관통하는 중심 객체인데 존재감이 약하다 → 항상 보이는 **여행 헤더**로 승격.
- **공유가 커뮤니티 글쓰기 모달에 묻혀 있다.** 일정 완성의 자연스러운 종착점인데 별도 화면으로 이동해야 함 → 일정 화면 안에서 "공유" 한 번으로.
- **밀도 일관성**: design_system이 workspace(dense)/content(airy) 2단을 정의하나, 실제론 탐색과 일정이 서로 다른 트리·서로 다른 인터랙션이라 "같은 언어"로 안 느껴짐.

## 2. IA · 내비게이션 재정리

### 2.1 핵심 결정 — **탐색과 일정을 하나의 "여행 작업실(Planner)"로 통합한다**

> 가장 큰 변화. 별개의 `/explore`와 `/schedule`을 **단일 작업 공간 `/plan`** 으로 합치고, 그 안에서 **모드 전환(탐색 ↔ 정리)** 으로 본다. 지도는 두 모드의 공통 배경으로 항상 켜져 있고, 우측 패널만 바뀐다.

- **[근거: 학습성·일관성·효율]** "찾기"와 "짜기"는 동일 작업의 두 국면이지 두 페이지가 아니다. 화면을 안 떠나면 "담은 게 일정 어디 들어갔나"를 즉시 본다. 컨텍스트 전환 0회.

### 2.2 새 사이트맵

```
TripCraft
│
├─ /                  랜딩 (제품 소개 + "여행 시작하기")
├─ /discover          탐색 전용 진입 (비회원도 자유 탐색 · 지도+필터+리스트)   ← 가벼운 둘러보기
│
├─ /plan/:tripId      ★여행 작업실 (통합 워크스페이스) — 시그니처
│      ├─ mode: explore   (지도 상시 배경 + 검색/필터/결과, 장소 담기)
│      └─ mode: organize  (24h 격자 타임테이블 유지 + 토글 지도, 클릭/드래그 배치·정밀 시간)
│      └─ 공유 = 작업실 안에서 바로 (게시판으로 push)
│
├─ /trips             내 여행들 (카드 그리드 · 새 여행 · 달력 토글)          ← /calendar 흡수
│
├─ /community         커뮤니티 (공유 일정 피드 · 공지 · 상세)
│      └─ /community/:id
│
├─ /me                마이페이지 (내정보 · 방문지도 · 내글 · 북마크 · 좋아요)
│
└─ /admin             관리자 (공지 CRUD · 글 관리)
```

**전역 내비게이션(상단 GNB) — 4개로 압축:**

```
[TripCraft]   탐색   내 여행   커뮤니티        [현재 여행 칩 ▾]  [검색]  [프로필 ▾]
              ────   ──────   ──────          ───────────────
              자유   여행 관리  공유/영감        ★항상 보이는 현재 여행
```

- 기존 GNB의 "탐색 / 내 일정 / 여행일지 / 커뮤니티 / 마이"(5탭) → **탐색 / 내 여행 / 커뮤니티(3탭) + 현재 여행 칩 + 프로필**.
  - **[근거: 학습성]** 5개 1차 메뉴는 초보가 "어디서 시작?"을 못 정함. "탐색(찾기) / 내 여행(짜기·관리) / 커뮤니티(공유·영감)"는 제품 본질의 3단계와 1:1.
  - **"탐색" 탭은 단일 라벨이되 맥락 라우팅**(§2.4): 현재 여행 있으면 `/plan` explore 모드, 없으면 `/discover`. "탐색"이 두 군데로 갈라져 보이지 않게 한다.
- **여행일지(`/calendar`)는 독립 탭에서 빼고 `/trips` 안 "달력 보기" 토글로 흡수.** "내 여행 카드 그리드"와 "달력"은 같은 데이터의 두 표현일 뿐.
  - **[근거: 정보밀도·일관성]** 같은 데이터에 1차 탭을 두 개 줄 이유가 없다.
- **현재 여행 칩**: 어느 화면에 있든 우상단에 `[제주 3박4일 ▾]`. 클릭 = 여행 전환/새 여행. 칩 자체가 `/plan/:id`로의 바로가기.
  - **[근거: 일관성·피드백]** 전 화면을 관통하는 중심 객체(activeTrip)에 **고정 자리**를 준다. "지금 무슨 여행 작업 중"이 항상 보임.

### 2.3 권한 × 화면 (브리프 §2 유지)
| | 비회원 | 회원 | 관리자 |
|---|---|---|---|
| `/discover` 탐색·`/community` 열람 | ✅ | ✅ | ✅ |
| `/plan` 담기·정리·저장·공유, `/trips`, `/me`, AI챗봇 | 🔒→로그인 유도 | ✅ | ✅ |
| 공지 CRUD·글 관리 | — | — | ✅ |

- 비회원이 `/discover`에서 장소를 "담기" 누르면 → **로그인 유도 시트** + "지금 담은 장소는 로그인 후 첫 여행에 자동 추가" (작업 손실 0).
  - **[근거: 실수방지]** 비회원의 탐색 노력을 버리지 않는다.

### 2.4 IA 모호성 해소 — **"탐색이 두 군데"로 안 느끼게 (단일 탐색 경험 + 라우팅 규칙)**

> 문제: GNB "탐색" 탭과 `/plan`의 explore 모드가 **둘 다 지도+필터+장소 탐색**이라, 사용자가 "탐색이 두 곳인가?"라고 헷갈릴 수 있다. → **"탐색은 하나의 경험이고, 맥락(현재 여행 유무)에 따라 같은 화면이 다른 옷을 입는다"**로 정리한다.

**핵심 모델: 탐색 화면은 1개, 진입점(GNB 탐색)이 컨텍스트에 따라 라우팅된다.**

```
GNB "탐색" 클릭
   │
   ├─ 현재 여행 있음(로그인 + activeTrip)  →  /plan/:tripId  (explore 모드, 담기→그 여행 보관함)
   │
   └─ 현재 여행 없음(비회원 또는 여행 0개) →  /discover       (담기→로그인/새 여행 유도 시트)
```

- **둘은 다른 화면이 아니라 "같은 탐색 UI의 두 상태"다.** 레이아웃·필터·카드·지도 동기화는 100% 동일. 차이는 **우측 패널(담기 대상)뿐** — `/plan`은 "현재 여행 보관함", `/discover`는 "로그인하면 담김" 안내.
  - **[근거: 일관성·학습성]** 탐색 인터랙션을 한 번만 학습하면 두 진입점 모두에서 통한다. "두 개의 탐색"이 아니라 "하나의 탐색, 두 맥락".
- **현재 여행 칩이 모드 표지 역할.** `/plan` explore에선 칩이 `[제주 3박4일 ▾]`로 채워져 "이 탐색은 제주 여행에 담는 중"임을 항상 알린다. `/discover`에선 칩 자리에 `[+ 새 여행]`이 떠 "아직 담을 여행이 없다"를 알린다.
  - **[근거: 피드백]** "지금 어디에 담고 있나"가 화면 상단에 상시 표시 → 길 잃음 방지.
- **GNB 라벨은 "탐색" 하나만.** "탐색"과 "둘러보기"처럼 동의어 2개를 1차 메뉴에 두지 않는다. 비회원이 `/discover`에서 담기→로그인→여행 생성하면, **이후 같은 "탐색" 탭이 자동으로 `/plan` explore로 연결**돼 끊김 없이 이어진다.
  - **[근거: 일관성·실수방지]** 라벨 중복 제거 + 로그인 전후 경로 연속성.
- `/discover`는 **딥링크·SEO·비회원 영업용 공개 URL**로서 의미가 있다(공유 링크로 진입 가능). GNB는 그 URL을 직접 노출하지 않고 위 규칙으로 자동 분기한다.

## 3. 핵심 플로우 재설계 — "더 적은 단계로"

### 3.1 Before (현재) — 7+ 스텝, 화면 2회 전환
```
새 일정 모달(제목·날짜·인원) → 탐색 화면 이동 → 필터로 찾기 → 카드 "추가+"
 → 일정 화면으로 전환 → 후보군 3단 트리 펼치기 → 24h 격자에 드래그 → 리사이즈로 시간조정
 → 이동 pill 자동삽입 대기 → 커뮤니티 이동 → 글쓰기 모달에서 일정 연결 → 공유
```

### 3.2 After (재설계) — 한 화면 안에서, 전환 0회
```
[여행 시작] 목적지 한 줄 입력("제주, 3박4일")        ← 모달 대신 1줄. 날짜는 나중에 조정 가능
   ↓ 바로 /plan 작업실, explore 모드로 진입
[탐색 모드] 지도+리스트에서 장소 카드 [담기]         ← 담으면 우측 '미배치 보관함'에 즉시 쌓임(피드백)
   ↓ "정리하기" 토글 (또는 보관함이 차면 자연 유도)
[정리 모드] 24h 격자에 카드 배치 — 보관함 카드 [+ Day·시간]  ← 클릭이 기본, 드래그는 보너스
           시작시간 = 직전 블록+이동시간 자동 제안(수정 가능)  ← 정밀 격자·리사이즈는 그대로
   ↓ (이동시간·동선은 자동 계산, 정밀 시간조정은 격자에서)
[공유] 작업실 우상단 [공유] → 제목만 확인 → 게시        ← 같은 화면, 모달 1개
```

- **모달 → 1줄 시작**: 일정 생성 모달의 필수입력(제목·출발일·귀환일)을 **"어디로, 며칠"** 한 줄로. 제목 자동생성("제주 3박4일"), 날짜는 작업실 헤더에서 언제든 조정.
  - **[근거: 효율·학습성]** 첫 진입 장벽을 모달에서 인풋 1개로. "일단 시작"이 쉬워야 한다.
- **담기 = 즉시 보관함에 쌓임(같은 화면 우측)**: 화면 전환 없이 "담겼다"가 보인다.
  - **[근거: 피드백]** 행동→결과가 같은 시야 안에.
- **배치 = 격자에 클릭으로 넣기**(시작시간 자동 제안): 24h 격자는 유지하되, 보관함 카드의 `[+ Day·시간]`을 누르면 **직전 블록+이동시간으로 시작시간이 자동 제안**돼 격자에 꽂힌다. 드래그·정밀 리사이즈는 그대로 쓸 수 있는 보너스.
  - **[근거: 실수방지·효율]** 드래그가 유일 경로일 때의 막힘 제거. 정밀 시간 입력은 "의무"가 아니라 "원하면" 할 수 있는 옵션으로.
- **공유 = 작업실 안 버튼**: 완성의 종착점을 같은 화면에 둔다.
  - **[근거: 효율]** 커뮤니티로 이동 후 일정 재선택하는 왕복 제거.

### 5.2 현재 대비 핵심 변화 — **격자는 유지, 인터랙션만 개선**

1. **24h 격자·정밀 시간·리사이즈 핸들 = 유지(강점 보존).** 요일 컬럼 × 시간축, 블록의 픽셀 위치(시작시간)·높이(체류), 30분 스냅 리사이즈, 블록 사이 이동 pill — **현 구현 그대로**. 정밀 제어·이동시간 시각화가 이 제품의 차별점이라 버리지 않는다.
   - **[근거: 일관성·정보밀도]** 시간 밀도·과밀·빈 시간대를 한눈에 보는 격자의 강점은 대체 불가. 시연·채점 포인트 보존.
2. **★클릭 추가를 기본 경로로 추가(드래그는 보너스).** 보관함 카드의 `[+ 배치 ▾]`을 누르면 → ⓐ "Day·시간 선택" 작은 시트 또는 ⓑ "Day 끝에 자동". **시작시간은 직전 블록 종료 + 이동시간으로 자동 제안(수정 가능).** 드래그를 못 해도 격자를 100% 채울 수 있다.
   - **[근거: 실수방지·효율]** 드래그가 유일 경로일 때의 막힘 제거. 키보드·터치 접근성↑. 자동 시작시간 제안으로 "빈 시간 직접 잡기" 부담↓.
3. **★삭제 = "빼기 = 미배치 보관함 복귀"**(손실 0). 블록 `[⋯]` 메뉴의 "빼기"를 누르면 격자에서 사라지지 않고 **보관함으로 되돌아간다.** "사이드바(삭제 존)로 드래그"의 오발 제거.
   - **[근거: 실수방지]** 실수로 빼도 복구 1클릭. 영구 삭제 아님 = 안심하고 정리.
4. **★이동 pill = 인라인 상태.** 카드 사이 pill이 ⓐ 계산 중=인라인 스피너, ⓑ 실패=`경로 없음 + 직선거리 ~분(추정)` 칩, ⓒ 클릭=수단 전환(🚌/🚶/🚗)을 **모달 없이 인라인 펼침**. 자동 계산·삽입 로직은 현행 유지.
   - **[근거: 피드백·정보밀도]** 비동기 계산 상태가 보임. 모달 왕복 제거. AC-08-03(경로 없음) 시각화.
5. **★미배치 보관함 = 플랫 + 분류색.** 현재의 "도시 > 시군구 > 분류" 3단 트리(`city-header>sigungu-header>cat-header`)를 **폐기**하고, 미배치 카드를 **플랫 1열 + 좌측 분류 ink 바**로. 탐색 결과 리스트와 **같은 언어**.
   - **[근거: 일관성·효율]** 탐색에서 보던 카드 모양 그대로 → 학습 1회. 펼치기 클릭 0.
6. **★Day 요약(활동·이동 합 → 과밀 경고).** Day 탭 옆/하단에 "Day1 활동 5h · 이동 38분". 합이 과하면 경고색 "다소 빡빡해요".
   - **[근거: 피드백·실수방지]** "무리한 일정"을 사전 경고. 격자만 봐선 총량이 안 보임.
7. **★헤더 = 현재여행 칩 · [공유] · 자동저장 "저장됨✓".** activeTrip을 toolbar select에서 **헤더 칩으로 승격**. 공유는 커뮤니티로 이동하지 않고 **이 화면 [공유] 버튼 한 번**으로(제목 확인 시트→게시). 자동저장은 "저장 중…→저장됨✓"(debounce).
   - **[근거: 일관성·효율·피드백]** 중심 객체 고정 자리. 완성→공유 왕복 제거. AC-09(자동저장).

### 5.3 "장소를 격자에 넣는 행위" — 더 쉬운 인터랙션 정의 (격자 유지 전제)

| 방식 | 동작 | 누구를 위한 것 |
|---|---|---|
| **① 클릭 추가(기본·신규)** | 보관함 카드 `[+ 배치 ▾]` → "Day·시간 선택" 또는 "Day 끝에 자동". 시작시간=직전 블록+이동시간 자동 제안(수정 가능) | 모두(드래그 못 해도 OK) — **[근거: 실수방지·효율]** |
| **② 드래그·드롭(보너스·유지)** | 카드를 격자 시간 위치로 드래그, 핸들로 체류 리사이즈 | 마우스 능숙자 정밀·빠른 배치 |
| **③ 자동 배치(초안)** | "미배치 N곳 자동 배치 →" → 지도 거리 기반 효율 순서로 격자에 채움(수정 가능) | 빠르게 초안 — **[근거: 효율·학습성]** |

- 핵심: **드래그·리사이즈는 "정밀하게 하고 싶을 때의 빠른 길"로 보존**하되, **클릭 추가가 항상 가능한 기본 경로**라 드래그 실패 시에도 막히지 않는다. 현재는 드래그가 유일 경로라 실패=막힘.

### 5.4 주요 상태

| 상태 | 화면 |
|---|---|
| **빈(미배치 0·블록 0)** | 격자는 유지(빈 시간 그리드)하되, 그 위에 오버레이 "탐색에서 가고 싶은 곳을 담아오세요 [탐색 모드로]". 보관함 비면 명확한 1버튼. **[근거: 학습성]** 빈 격자에 "다음 행동" 안내. |
| **보관함 차고 격자 비었을 때** | 격자 상단 배너 "미배치 4곳을 자동 배치 →" 큰 버튼 + 각 카드 `[+ 배치]`. **[근거: 효율]** |
| **이동 계산 중** | 해당 pill 자리 인라인 스피너 "계산 중", 나머지 격자·블록 정상 조작(논블로킹). **[근거: 피드백·실수방지]** |
| **이동 계산 실패** | pill = "경로 없음" 회색 칩 + "직선거리 ~분(추정)" 보조. 블록·저장은 정상. **[근거: 실수방지]** AC-08-03 충족. |
| **자동저장** | 헤더 "저장 중…→저장됨✓"(debounce). **[근거: 피드백]** AC-09. |

### 5.5 지도-격자 폭 충돌 해법 — **organize 모드의 지도 배치**

> 24h 격자는 가로 폭을 많이 먹어, explore처럼 지도를 상시 배경에 깔면 격자가 좁아진다. organize 모드에서 지도를 어디에 둘지 3안을 검토하고 **A안(토글 가능한 우측 패널)을 채택**한다.

| 안 | 설명 | 장점 | 단점 | 판정 |
|---|---|---|---|---|
| **A. 토글 우측 패널(채택)** | 기본은 보관함+격자가 가로 풀폭. 헤더 `[지도 ▸]`로 우측에서 슬라이드인(격자 폭 일시 축소), `[닫기 ◂]`로 복귀. | 격자가 평소 넓게·동선 볼 땐 즉시. 토글 상태 세션 기억. explore의 "상시 지도"와 명확히 구분돼 모드 정체성↑ | 지도가 상시는 아님(한 번 더 클릭) | **채택** |
| **B. "지도" 서브탭** | organize 안에 [격자][지도] 서브탭. 지도 탭=동선 풀스크린. | 각자 풀폭, 단순 | 격자↔동선 동시에 못 봄(배치하며 동선 확인 불가) | 보조 |
| **C. 격자/지도 좌우 분할 고정** | 항상 격자50:지도50. | 항상 둘 다 보임 | 격자가 늘 절반→시간 가독성↓(격자 강점 훼손) | 탈락 |

- **A안 채택 근거 [정보밀도·효율]**: 격자(정밀 시간)는 평소 넓을수록 좋고, 지도(동선)는 "배치 후 확인" 빈도라 토글이 맞다. **explore=지도 상시 배경, organize=격자 우선+지도 토글**로 두 모드의 시각 정체성을 분리해 "지금 찾는 중 vs 짜는 중"을 한눈에 구분.
- 지도 ON 시: ①②③ 번호 마커(Day색)·구간 경로선(대중교통=노선색/도보=초록 점선)은 현행 시각화 유지. 격자에서 블록 hover→지도 마커 강조(양방향 동기화).

## 6. 비주얼 방향 — Warm Editorial Workspace 유지 + 미세 보정

> design_system.md의 **Warm Editorial Workspace**(따뜻한 종이 베이스 `#F5F4F0` + 퍼플 액센트 `#534AB7`, 위계는 타이포·색은 절제, 테두리보다 그림자, 라인 SVG 1세트, **분류 6색 전 화면 공통**)를 **그대로 출발점으로 채택**한다. 여행/에디토리얼 도메인에 적합하고, 정보 밀도 높은 작업 화면에서 차가운 회색 대비 피로가 낮다.

### 6.1 채택 근거 (대안 검토 후 유지)
- 후보 대안 ❶ 다크 데이터 대시보드(`#0E1116`/네온) — 지도+사진 콘텐츠가 죽고 여행감 상실. **탈락**.
- 후보 대안 ❷ 화이트+블루 SaaS — 무난하나 "에디토리얼 여행"의 개성 없음(AI슬롭 위험). **탈락**.
- → **Warm Editorial 유지가 도메인 적합·차별성 모두 우위.** 단 아래 보정.

### 6.2 v2 보정 사항 (사용성 근거 포함)
1. **분류 6색 = 전 화면 단일 신호로 격상.** 지도 핀·결과 카드 좌측 바·보관함 카드·**격자 이벤트 블록**·달력 띠에서 **동일 ink 색**. 지금도 토큰은 있으나 화면마다 적용 편차가 있다 → "이 색=이 분류"를 깨지 않게 강제.
   - **[근거: 일관성·학습성]** 색 하나로 분류를 인지 → 텍스트 안 읽어도 됨. (`--cat-*` tint=카드/배경, ink=핀·블록 규칙 고정)
2. **Day 색 ≠ 분류 색 충돌 방지.** 지도 동선의 번호 마커·경로선은 **Day별 뉴트럴 강조**(퍼플 명도 스텝)로, 분류 6색과 레이어를 분리.
   - **[근거: 일관성]** "분류색"과 "Day색"이 안 싸우게.
3. **밀도 2단 유지·정제**: 작업실(/plan)=dense(타이포 13~14, space 8~12), 콘텐츠(/community,/trips,/me)=airy(15~16, space 16~24). **같은 토큰**에서 파생.
   - **[근거: 정보밀도]** 작업은 빽빽이, 읽기는 넉넉히 — 같은 언어로.
4. **모션은 고임팩트 순간에만**: ⓐ 담기→보관함 카드 fly-in(행동 확인), ⓑ 모드 전환(explore↔organize) 우측 패널 슬라이드+지도 줌 변화, ⓒ 이동시간 자동 삽입 시 pill 페이드인. 그 외 정적.
   - **[근거: 피드백]** 결과를 눈으로 확인시키되 산만하지 않게.
5. **타이포 위계로 영역 구분(색박스 금지)**: 패널 제목 `--text-xl(18)`, 카드 제목 `--text-md(15)`, 메타 `--text-xs(12)`. 헤어라인 남발 대신 `--shadow-sm/md`로 카드 부양.
   - **[근거: 정보밀도·일관성]** design_system 원칙 1·2 그대로.
6. **아이콘 라인 SVG 1세트, 이모지 금지.** (현재 코드에 `🗺 ▶ ✕ 🚌🚗🚶` 잔존 — 확산 시 Icon 컴포넌트로 일괄 치환 대상)
   - **[근거: 일관성]**

### 9.2 핵심 요소
- **§4와 동일**: 칩 필터(지역/분류/⭐) + 플랫 결과 리스트 + 리스트↔핀 양방향 hover/선택 + 미니카드→슬라이드 상세.
- **우측 = "둘러보기 보관함"**: 담기 누른 장소를 **세션(localStorage)** 에 임시 적재. 상단에 "🔒 로그인하면 담은 곳이 첫 여행으로" 안내.
  - **[근거: 실수방지]** 비회원의 탐색 노력 손실 0. 로그인 후 첫 여행에 자동 합류.
- **GNB는 2탭(탐색·커뮤니티) + [+ 새 여행] + [로그인]**: 비회원이 못 쓰는 "내 여행"은 숨기되, 칩 자리에 `[+ 새 여행]`을 둬 가입 경로 노출.
  - **[근거: 학습성]** 권한 밖 메뉴를 회색으로 두지 않고 아예 숨겨 혼란↓.

### 10.3 핵심 요소 · 결정

  - **[근거: 정보밀도·일관성]** 같은 데이터에 탭 2개 줄 이유 없음(§2.2 여행일지 흡수).

## 13. AI 챗봇 패널 (상세 내 동거)

> 독립 화면이 아니라 **장소 상세 슬라이드인 안에 동거**하는 패널(§4 미니카드→상세). 주변 추천을 받아 지도 핀으로 연결. overlay 밀도.

### 13.1 레이아웃 (상세 슬라이드인 하단/탭)
```
┌── 장소 상세 (슬라이드인) ──────────────────┐
│ [히어로 이미지]                            │
│ 성산일출봉   관광지 · 서귀포                │
│ 주소·전화·운영시간 …                       │
│ ── [ 소개 ][ AI 도우미 ] ──────────────────│  ← 탭으로 챗봇 동거(상세 컨텍스트 유지)
│ ┌────────────────────────────────────────┐│
│ │ 🤖 성산일출봉 주변에서 더 가볼 만한 곳은? ││
│ │ ─ 가까운 카페 3곳 추천드려요             ││
│ │   ① 카페한라 (도보 8분)  [지도에 표시]    ││  ← 추천→클릭 시 지도 핀 점등+포커스
│ │   ② …                   [담기+]         ││  ← 추천 장소를 바로 보관함에
│ │ ─────────────────────────────────────── ││

│ └────────────────────────────────────────┘│
└────────────────────────────────────────────┘
```

### 13.2 핵심 요소 · 결정
- **상세 탭 안에 동거**([소개]/[AI 도우미]): 별도 화면 이동 없이 그 장소 컨텍스트로 질문.
  - **[근거: 효율·정보밀도]** "이 장소 주변" 질문이 상세를 떠나지 않고 됨.
- **주변추천 → 지도핀 연결**: 추천 항목 `[지도에 표시]` → 해당 핀 점등·지도 포커스. `[담기+]` → 보관함 즉시 적재.
  - **[근거: 피드백]** 텍스트 추천이 지도·일정과 즉시 연결(말→행동).
- **뒤로가기 컨텍스트 복원(AC-13-05)**: 추천 핀을 보러 지도로 갔다가 돌아오면 **대화 맥락·스크롤 유지**.
  - **[근거: 실수방지]** 대화 손실 방지.

| 상태 | 화면 |
|---|---|
| **초기** | "이 장소에 대해 물어보거나, 주변 추천을 받아보세요" + 예시 칩("주변 맛집","가는 법"). **[근거: 학습성]** |
