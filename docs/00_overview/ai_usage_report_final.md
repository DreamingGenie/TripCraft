# AI 사용 보고서 — TripCraft (final)

> **버전**: final (2026.06)
> **작성자**: 전진 · 송정기
> **범위**: ① 개발 과정의 AI 활용(Claude Code) ② 제품에 탑재한 AI 기능(Spring AI 챗봇)

---

## 1. 개요

TripCraft는 AI를 **두 층위**에서 활용했다.
1. **개발 보조** — Claude Code(에이전트형 코딩 도구)를 협업 파트너로 두고 설계·구현·문서화·리뷰를 가속.
2. **제품 기능** — 관광지 상세에 **대화형 AI 도우미**(Spring AI 기반)를 직접 탑재해 사용자 가치를 제공.

AI는 보조 수단이며, **모든 산출물은 사람이 코드·빌드·실행으로 검증**하는 것을 원칙으로 했다.

---

## 2. 개발 과정의 AI 활용 (Claude Code)

### 2-1. 에이전트 협업 환경 구축
프로젝트 첫날(2026-05-22 작업 일지)부터 **AI 에이전트가 프로젝트 규칙을 이해하도록** 컨텍스트를 구조화했다.

| 파일 | 역할 |
|------|------|
| `CLAUDE.md` | 기술 스택, 글로벌 규칙(MyBatis `#{}` 강제·서버 권한 검증·공통 응답 형식), Git/MR 컨벤션, docs 정책 |
| `.claude/member.md` | 회원·인증·보안 도메인 컨텍스트 |
| `.claude/attraction.md` | 관광지 조회·TourAPI 수집 전략 |
| `.claude/plan.md` | 일정·이동시간 비즈니스 로직, 캐시 레벨 |
| `.claude/community.md` | 커뮤니티·공지 권한 규칙 |
| `.claude/chat.md` | AI 챗봇·주변 추천 |

→ 효과: 매 작업마다 규칙을 다시 설명할 필요 없이, AI가 **프로젝트 컨벤션을 지킨 코드/문서**를 생성.

### 2-2. 개발 단계별 활용
| 단계 | 활용 |
|------|------|
| **설계** | DB 스키마 설계 논의, ERD/요구사항 정리, 동시성 정책 선택지 비교 |
| **구현** | 도메인 보일러플레이트(컨트롤러·매퍼·DTO), 외부 API 파싱, Vue 컴포넌트 |
| **디버깅** | rate limit 빈 응답 원인 규명, grab release 누락 버그 추적, 이중 인코딩 이슈 |
| **리뷰·문서화** | MR 설명, 기술 노트, 본 산출물 문서 작성 |

### 2-3. 대표 활용 사례
- **외부 API 오케스트레이션**: ODsay 도시간 경로의 접근 구간 합성·다층 캐시 설계를 AI와 반복 검토 (`transit_external_api.md`).
- **동시성 의사결정**: "무엇을 충돌로 볼 것인가"를 행렬로 정리하고, 잘못 진단된 race는 **기계적으로 구현하지 않고 정정** (`collab_concurrency_decision.md`).
- **성능 개선**: 게시글 목록 N+1을 파생 테이블 JOIN으로 전환 (`perf_n1_profile_image.md`).

### 2-4. 검증 원칙 (중요)
AI 산출물을 그대로 신뢰하지 않고 항상 검증했다.
- 백엔드 `compileJava`, 프론트 `vite build` 통과 확인.
- 권한·SQL 등 민감 로직은 코드/스키마와 1:1 대조.
- AI의 분석이 틀린 경우(예: `computeIfAbsent` 원자성 오진) **사람이 재검증해 정정**.

> 교훈: AI는 **속도와 탐색 범위**를 키우지만, **결정과 검증의 책임은 개발자**에게 있다.

## 3. 제품에 탑재한 AI 기능 — 관광지 AI 챗봇

관광지 상세에서 자연어로 묻고 주변 추천까지 받는 **대화형 도우미**.

### 3-1. 아키텍처
- **프레임워크**: Spring AI (`spring-ai-starter-model-openai`, BOM 1.0.6)
- **모델/게이트웨이**: gms(OpenAI 호환 프록시, `gms.ssafy.io`) 경유 `gpt-4.1`
- **엔드포인트**: `POST /api/attractions/{id}/chat { message, conversationId? }` → `{ reply, conversationId, nearby[] }`
- **인증**: AI 토큰 비용 때문에 **회원 전용**(GET만 공개라 POST는 자동 인증 대상)

### 3-2. 핵심 설계
- **컨텍스트 주입**(`buildSystemPrompt`): 해당 관광지의 이름·분류·지역·주소·이용정보·소개를 한국어 시스템 프롬프트로 직렬화(HTML 태그 제거). → 일반 LLM이 아니라 **"이 장소를 아는" 도우미**가 됨.
- **주변 추천**: 좌표 기준 반경 **3km·최대 8곳**을 `ST_Distance_Sphere`(bounding box 1차 필터 + 구면 거리 정렬)로 뽑아 프롬프트에 주입. 지침에 **"목록에 없는 곳은 지어내지 말 것"**(환각 방지)을 명시.
- **멀티턴**: `MessageChatMemoryAdvisor` + InMemory `ChatMemory`(기본 20메시지). `conversationId`로 대화 식별(미지정 시 서버가 UUID 발급).

### 3-3. 프론트 UX
- `AttractionChat.vue` — 상세 패널 내 접이식 챗 UI.
- `stores/attractionChat.js` — **관광지별 대화·아코디언 상태 보존**(패널 리마운트·네비게이션에도 유지).

### 3-4. 한계
- 대화 메모리가 인메모리라 **서버 재시작 시 소멸**(영속화 미적용).
- 주변 추천은 DB 적재 관광지 한정(실시간 외부 검색 아님).

## 5. 윤리·주의

- AI 생성 코드/문서는 **검토 후 채택**, 보안·권한 로직은 사람이 최종 책임.
- 제품 챗봇은 환각 방지 지침·DB 근거(주변 목록)로 **사실 기반 응답**을 유도.
- 외부 AI 게이트웨이(gms) 키는 `.env`로 관리(코드·저장소 미노출).

> 부록(선택): 실제 사용한 대표 프롬프트·세션 캡처는 발표 자료에 별첨 가능. 작업 일지(`docs/04_logs/*`)가 세션 단위 기록을 보유.
