## 2. 기획 배경

### 2-1. 문제 인식

국내 여행을 계획할 때 사용자는 **최소 4~5개의 서로 다른 서비스**를 동시에 열어두고 작업해야 한다.

가고 싶은 관광지를 포털 지도로 검색하고, 운영 시간과 예약 가능 여부는 개별 사이트에서 확인하며, 장소 사이의 이동 수단과 소요 시간은 대중교통 앱으로 따로 계산하고, 숙소는 예약 플랫폼에서 비교한다. 일행이 있다면 이 모든 과정을 메신저와 노션, 공유 메모 등 여러 채널을 오가며 맞춰가야 한다.

결국 "여행 계획을 세우는 일" 자체가 피로하고 복잡한 작업이 된다. 세심하게 짠 일정도 막상 현장에서 이동 시간이 예상과 달라지면 전체 흐름이 무너지는 경우가 빈번하다.

### 2-2. 핵심 불편함

- **정보의 분산**: 관광지·이동·숙소 정보가 모두 다른 서비스에 흩어져 있어 통합된 뷰를 얻을 수 없다.
- **수동 이동 시간 계산**: 장소가 바뀔 때마다 이동 시간을 직접 검색하고 일정에 수기로 반영해야 한다.
- **일정의 경직성**: 예약 장소의 방문 시간이 고정되어 있어도, 전후 일정과의 연계를 자동으로 재배치해 주는 도구가 없다.
- **협업의 불편함**: 노션·스프레드시트 등 범용 도구는 여행 일정 특유의 시간·장소 구조를 표현하기 어렵고, 일행과 실시간으로 논의하며 수정하기에 적합하지 않다.

### 2-3. 해결 방향

TripCraft는 이 네 가지 불편함을 하나의 플랫폼 안에서 해소하는 것을 목표로 한다.

**장소를 지정하면 이동 경로와 소요 시간이 자동으로 계산**되고, 예약이 확정된 장소가 있다면 그 방문 시간을 기준으로 전후 일정이 자동 재배치된다. 일행과는 블록 형태의 시간표를 함께 편집하며 실시간으로 일정을 논의할 수 있다. 관광지 정보는 한국관광공사 공공 API를 기반으로 한 자체 DB에서 제공하여 별도 서비스로 이탈하지 않아도 된다.

## 3. 서비스 목표

### 3-1. 핵심 가치 (Core Value)

| 가치 | 설명 |
|------|------|
| **통합(Integration)** | 관광지 탐색 → 일정 설계 → 이동 계산 → 공유까지 하나의 플로우로 |
| **자동화(Automation)** | 장소를 추가하는 것만으로 이동 시간과 동선이 자동 반영 |
| **협업(Collaboration)** | 여행 일정 특화 UI로 일행과 실시간 공동 편집 |

### 3-2. 목표 사용자

- 국내 여행지를 직접 조사하고 일정을 짜는 **자유여행 선호자**
- 2인 이상이 함께 여행 계획을 세우는 **소그룹 여행자**
- 촘촘한 일정보다는 유연한 플로우를 선호하지만, **기본 동선은 미리 잡아두고 싶은** 여행자

## 4. MVP 기능 정의

> MVP(Minimum Viable Product)는 핵심 사용자 가치를 검증할 수 있는 최소 기능 집합입니다.  
> 개발 리소스(2인, 8주)를 고려하여 **Phase 1 (MVP)** 과 **Phase 2 (고도화)** 로 나눕니다.

### Phase 1 — MVP (필수 구현)

| ID | 기능 영역 | 기능명 | 설명 |
|----|----------|--------|------|
| F01 | 관광지 | 관광 DB 구축 | 한국관광공사 국문 관광정보 API로 관광지 데이터 수집 및 저장 |
| F02 | 관광지 | 지역별 조회 | 시도·시군구 단위 필터링으로 관광지 목록 제공 |
| F03 | 관광지 | 카테고리별 조회 | 관광지·숙박·음식점·문화시설 등 콘텐츠 타입별 조회 |
| F04 | 관광지 | 즐겨찾기 | 관심 장소를 개인 즐겨찾기에 저장·관리 |
| F05 | 일정 | 블록형 시간표 | 드래그 앤 드롭으로 장소 블록을 시간표에 배치 |
| F06 | 일정 | 이동 시간 자동 계산 | 장소 간 대중교통·도보 이동 시간 자동 삽입 |
| F07 | 일정 | 일정 저장·불러오기 | 작성한 여행 일정을 저장하고 재편집 |
| F08 | 커뮤니티 | 여행 일정 공유게시판 | 완성된 일정을 게시판에 공유, 다른 사용자의 일정 열람 |
| F09 | 커뮤니티 | 공지사항 | 관리자 공지 CRUD |
| F10 | 회원 | 회원 관리 | 회원가입·로그인·로그아웃·정보수정·탈퇴 |

### Phase 2 — 고도화 (MVP 안정화 후 추가)

| ID | 기능명 | 설명 | 비고 |
|----|--------|------|------|
| G01 | 실시간 공동 편집 | WebSocket 기반 다중 사용자 동시 편집 | 기술 복잡도 높음 — Phase 2 권장 |
| G02 | 예약 연동 | 외부 숙소 및 관광지, 식당 예약 링크 또는 API 연동 | |

## 5. 기술 스택 (예상)

### Backend

| 항목 | 기술 | 선택 이유 |
|------|------|---------|
| 언어 | Java 21 | 기존 프로젝트 연속성 |
| 프레임워크 | Spring Boot 3.x | 자동 구성, 생태계 |
| ORM | JPA | 편리 |
| DB | MySQL 8.0 | 기존 환경 유지 |
| 빌드 | Gradle | Maven 대비 빌드 속도·유연성 |
| 인증 | Spring Security + JWT | 세션 → 토큰 방식으로 전환 (API 확장성) |
| 실시간 (Phase 2) | Spring WebSocket + STOMP | |

### Frontend

| 항목 | 기술 | 선택 이유 |
|------|------|---------|
| 프레임워크 | Vue.js 3 | 드래그 앤 드롭 컴포넌트 생태계 풍부 |
| 지도 | Kakao Maps API | 국내 주소 정확도, 한국어 지원 |
| 대중교통 | ODsay API | 실제 대중교통 경로·소요시간 |
| 드래그 앤 드롭 | vue-draggable / dnd-kit | |
