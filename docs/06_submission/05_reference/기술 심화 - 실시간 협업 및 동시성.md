# 기술 심화 — 실시간 협업 및 동시성

> TripCraft 여행 일정 편집을 **여러 명이 동시에 드래그앤드롭으로 편집**하는 협업 기능에서 가장 많은 노력을 쏟은 세 가지 쟁점을 정리한다. 
>
> 1. **WebSocket(STOMP) 기반 편집 실시간 적용**
> 2. **낙관적 락(Optimistic Lock) 기반 동시성 제어**
> 3. **실시간 커서 트래킹 + 상대 좌표 동기화**(전송된 정보를 자기 화면 기준으로 환산)
>
> 각 주제를 **① 왜 / ② 어떻게 / ③ 어려움 / ④ 고민·의사결정** 순서로 기술한다.

---

## 0. 전체 그림 (한눈에)

```
[사용자 A 브라우저]                 [Spring 서버]                 [사용자 B 브라우저]
  드래그/리사이즈 ──REST──▶ TripServiceImpl(@Transactional)
                              ├ 낙관적 락(version)·grab·겹침 검사
                              ├ DB 반영
                              └ broadcast(TripEvent, seq) ──STOMP──▶ /topic/trip/{id}
                                                                       └▶ handleTripEvent → loadTrip
  마우스 이동 ──STOMP──▶ /app/trip/{id}/pointer
                          TripPresenceController(in-memory)
                          └ broadcastPresence ──STOMP──▶ /topic/trip/{id}/presence
                                                           └▶ 좌표 역환산 → 커서/ghost 렌더
```

- **두 채널 분리**가 핵심 설계다.
  - **편집(영속 데이터)**: REST로 쓰고 DB 트랜잭션 + 낙관적 락으로 정합성 보장 → STOMP로 "변경됨" 신호 broadcast.
  - **presence(커서/참가자, 휘발 데이터)**: DB를 안 거치고 서버 in-memory(`ConcurrentHashMap`)만 사용 → 고빈도(마우스 이동)라 가벼워야 함.
- 이 분리 덕분에 "블록 편집은 트랜잭션·락으로 엄격하게, 커서는 빠르고 느슨하게"라는 서로 다른 요구를 각각 최적화할 수 있었다.

핵심 파일
- 프론트: `frontend/src/stores/collab.js`(STOMP 연결), `frontend/src/components/ScheduleBoard.vue`(편집·커서 송수신), `frontend/src/composables/useCollabCursor.js`(좌표 환산), `frontend/src/config/collab.js`(튜닝 변수)
- 백엔드: `plan/service/TripServiceImpl.java`(편집·락), `plan/controller/TripPresenceController.java`(presence), `global/security/JwtChannelInterceptor.java`·`JwtHandshakeInterceptor.java`(WS 인증), `global/security/TripAccessVersion.java`(권한 캐시 무효화)

## 1. WebSocket(STOMP) 기반 편집 실시간 적용

### ① 왜 WebSocket(STOMP)인가
- **요구**: A가 블록을 옮기면 B 화면에 "즉시" 반영. 양방향·저지연·다자 브로드캐스트가 필요.
- **대안 비교**
  - *폴링*: 주기적 조회 → 지연·트래픽 낭비, "실시간" 체감 부족.
  - *SSE(Server-Sent Events)*: 서버→클라 단방향. 커서 전송(클라→서버)이 잦아 부적합.
  - *WebSocket*: 양방향·저지연. 채택.
- **STOMP를 얹은 이유**: 순수 WebSocket은 메시지 라우팅·구독 개념이 없다. STOMP는 `/topic/...` **pub/sub 구독**과 `/app/...` **메시지 매핑**을 제공해, "일정별 토픽 구독", "편집/커서 채널 분리"를 선언적으로 구현할 수 있다. Spring이 `@MessageMapping`·`SimpMessagingTemplate`로 1급 지원.
- **SockJS 폴백**: 네트워크/프록시가 WebSocket을 막을 때 HTTP 스트리밍으로 폴백(`sockjs-client`).

### ② 어떻게 구현했나
- **토픽 설계**
  - `/topic/trip/{id}` — 편집 이벤트(BLOCK_ADDED/MOVED/DELETED, CANDIDATE_ADDED/REMOVED, TRANSIT_RECALCULATED)
  - `/topic/trip/{id}/presence` — 커서/참가자 목록(PRESENCE_UPDATE)
  - `/app/trip/{id}/pointer` — 클라가 커서/grab/keepalive를 발행하는 목적지
- **인증**(쿠키 기반, 헤더 토큰 없이)
  - 핸드셰이크: `JwtHandshakeInterceptor`가 쿠키의 access_token을 검증해 세션 속성에 `memberId` 저장.
  - `JwtChannelInterceptor`: CONNECT 시 `memberId`로 `Principal` 설정. SUBSCRIBE/SEND 시 해당 trip 접근 권한 검증.
  - **권한 캐시**: 커서 전송은 매우 잦으므로 SEND마다 DB 조회하면 부담. 그래서 SUBSCRIBE/SEND 1회 검증 후 세션에 캐시. 단 협업자 제거/공유 변경 시 캐시가 낡지 않게 `TripAccessVersion`(일정별 세대 카운터)을 두고, 세대가 바뀌면 재검증한다.
- **편집 → 브로드캐스트 흐름**
  - 편집은 REST(`placeBlock`/`updateBlock`/`removeBlock`)로 처리하고, 성공 시 `broadcast(tripId, TripEvent.of(...))`로 변경을 알린다.
  - 수신측 `handleTripEvent`(`ScheduleBoard.vue`)가 이벤트 타입에 따라 `loadTrip()`(전체 재조회) 또는 `applyTransitUpdate()`(부분 패치) 수행.
  - **자기 이벤트 무시**: 이벤트에 `actorId`를 실어, 내가 일으킨 변경은 무시(내 화면은 이미 낙관적 UI로 반영됨).
  - **순서·중복 방어**: `TripEvent.seq`(일정별 `AtomicLong` 단조 증가)를 broadcast 시 스탬프. 수신측은 `seq ≤ 마지막 처리 seq`면 무시 → 역전·중복 이벤트로 인한 불필요한 재조회 방지.
- **재연결**: STOMP 자동재연결 대신 **수동 재연결**(`reconnectDelay:0` + 직접 `setTimeout`)로 제어. 재연결 성공 시 `onReconnect → loadTrip()`로 끊긴 동안 놓친 변경을 한 번에 재동기화.
- **트랜잭션 위생**: 이동시간 재계산(`recalculateTransitForDate`)은 외부 API(ODsay/TMap)를 호출하는데, 이를 편집 트랜잭션 **커밋 이후**(`TransactionSynchronizationManager.afterCommit`)로 미뤄, 외부 지연 동안 DB 커넥션을 잡지 않게 했다.

### ③ 어려움
- **인증을 어디서 거는가**: WebSocket은 일반 HTTP 필터 체인을 안 탄다. 핸드셰이크(쿠키)에서 인증하고, STOMP CONNECT/SUBSCRIBE/SEND 단계별로 권한을 나눠 거는 구조를 잡는 데 시행착오가 있었다.
- **전체 재조회 vs 부분 패치**: 블록 이벤트는 단순화를 위해 `loadTrip()` 전체 재조회로 갔는데, 낙관적 UI(내가 방금 옮긴 블록)와 겹쳐 깜빡임이 생길 수 있다. 빈도·복잡도를 따져 전체 재조회를 유지하되, 이동시간만은 부분 패치(`applyTransitUpdate`)로 처리.
- **이벤트 유실/순서**: 네트워크 순간 끊김 시 이벤트를 놓칠 수 있다. seq로 순서/중복은 방어하되, 유실분 재조회 전용 엔드포인트까지는 만들지 않고 "재연결 시 전체 재조회로 수렴"하는 단순 정책을 택했다(데이터 유실은 없음).

### ④ 고민·의사결정
- **broadcast 시점**: 현재 BLOCK_* broadcast는 트랜잭션 커밋 직전에 일어난다(수신측이 곧 재조회). 엄밀히는 "커밋 후 broadcast"가 더 정확하지만, 이벤트 순서·정합성 보강(seq 부여, afterCommit 재계산)을 우선하고 이 부분은 백로그로 남겼다.
- **부하 인식**: presence broadcast는 인원이 늘면 트래픽이 ~O(n²)로 증가한다(인원마다 커서 전송 × 전원에게 재전송). 2~5인을 전제로 두되, 확장 시 좌표 델타 전송·coalescing이 1순위 보강임을 문서화했다.

## 2. 낙관적 락(Optimistic Lock) 기반 동시성 제어

### ① 왜 낙관적 락인가
- **문제**: 처음엔 편집 경로가 `@Transactional` 단일 트랜잭션 + **락 없음**이었다. 두 명이 같은 블록을 동시에 옮기면 나중 요청이 먼저 요청을 조용히 덮어쓴다(Last-Write-Wins → 데이터 유실).
- **선택지 비교**

  | 방식 | 막는 것 | 비용 | 판단 |
  |------|---------|------|------|
  | 비관적 락(`SELECT … FOR UPDATE`) | 모든 경합 | 항상 락 점유, 드래그 UX와 안 맞음 | 과함 |
  | **낙관적 락(version)** | 같은 row 동시 수정 | 컬럼 1개 + 조건부 UPDATE | **채택** |
  | grab 서버 강제 | 드래그 중 블록 선제 차단 | 기존 인프라 호출만 | 보조 |
  | CRDT/OT(자동 병합) | 모든 충돌 자동 해소 | 아키텍처 전면 전환(수 주) | ROI 낮음 |

- **낙관적 락이 맞는 이유**: 우리 환경은 **충돌이 드물다**(2~5인 + 실시간 커서로 서로 인지). 비관적 락은 평소에도 손해지만 낙관적 락은 **평소 오버헤드 0, 실제 경합 때만 한 명이 거부**된다. 충돌 빈도가 낮을수록 유리.

### ② 어떻게 구현했나
- **version 컬럼**: `trip_block.version INT NOT NULL DEFAULT 0`.
- **조건부 UPDATE**: `updateWithVersion` — `UPDATE … , version = version+1 WHERE id = ? AND version = ?` → **affected rows 반환**. 0행이면 그 사이 누가 먼저 바꾼 것 → `409 CONFLICT`.
- **버전 왕복**: 클라가 블록을 읽을 때 받은 `version`(`BlockItem`)을 편집 요청(`BlockUpdateRequest`)에 동봉. 409를 받으면 토스트 + `loadTrip()`으로 최신 상태 재조회(낙관적 UI 자가 복구).
- **두 층 방어**
  - *grab(사전 차단)*: 누가 드래그 중인 블록을 다른 사람이 건드리면 `getGrabOwner` 검사로 즉시 409("○○님이 편집 중"). UX 친화적 선제 차단.
  - *낙관적 락(사후 안전망)*: grab을 안 거친 경로·grab이 풀린 순간까지 최종 정합성 보장.
- **충돌 기준 매트릭스**(이 설계의 핵심 — "무엇을 충돌로 볼 것인가")

  | 시나리오 | 메커니즘 | 결과 |
  |---|---|---|
  | 같은 블록 동시 이동/리사이즈 | version | 둘째 409 → 재조회·재시도 |
  | 드래그 중 블록을 타인이 수정 | grab 게이트 | 즉시 409 |
  | **서로 다른** 블록을 각자 다른 위치로 | 독립 row·독립 version | **충돌 없음**(정상 동시 처리) |
  | transit 재계산 ↔ 사용자 편집 | transit 전용 UPDATE(version 불변) | **충돌 없음** |
  | 삭제 vs 같은 블록 이동 | 이동의 versioned UPDATE가 0행 | 이동 측 409 |
  | 다른 블록을 **같은 시간대**에 겹침 | trip 행 `FOR UPDATE` + 겹침 검사 | 둘째 409 |

  **핵심 불변식**: `version`은 "같은 row"에만 작용 → 무관한 블록 편집·transit 재계산은 절대 오탐(false conflict)하지 않는다.
- **시간대 겹침 금지**(cross-row 제약): version으로는 못 막는다(다른 row). MySQL엔 배제 제약도 없어, 검사 직전 **`trip` 행을 `SELECT … FOR UPDATE`로 잠가** 같은 일정 편집을 직렬화했다. 둘이 동시에 겹치는 자리에 놓아도 둘째는 첫째 커밋 후 최신 상태로 검사 → 한쪽만 통과. 부수적으로 `display_order` MAX+1 race도 이 잠금 안에서 안전해졌다.
- **transit 분리**: 사용자 편집은 `updateWithVersion`(version+1), 시스템의 이동시간 재계산은 `updateTransitById`(transit 컬럼만, version·위치 불변)로 **물리적으로 다른 쿼리**로 나눴다. 이게 매트릭스 4행(오탐 없음)의 전제다.
- **권한 캐시 무효화**: 협업자 제거/역할·공유 변경 시 `TripAccessVersion.bump(tripId)` → STOMP 인터셉터가 다음 프레임에 권한 재검증.

### ③ 어려움 (실제로 부딪힌 버그들)
- **transit 재계산이 false-conflict를 유발**: 처음엔 재계산도 전체 UPDATE라 version을 올려, 사용자 편집 직후 시스템 재계산이 상대의 version을 무효화 → 멀쩡한 편집이 거부됐다. → transit 전용 UPDATE로 분리해 해결.
- **resize 후 재수정 시 단일 창에서도 409**(인상 깊었던 버그): `onResizeEnd`는 성공 후 `loadTrip` 재조회를 안 해 로컬 `ev.version`이 옛값 그대로 → 다음 수정에서 stale version → 0행 → 409. 협업도 아닌 한 창에서 재현됐다. → 성공 시 `ev.version = ev.version + 1`로 서버의 version+1과 동기화해 해결.
- **grab 잠금이 안 풀림**: grab을 표시용으로만 쓸 땐 안 보였는데, 서버 게이트로 승격하니 드러났다. 드래그 종료 시 프론트가 `targetBlockId:null`을 보내 서버 grabMap이 안 비워졌고, keepalive가 presence를 살려 stale evict도 안 돌아 **세션 내내 잠금 잔존** → 남이 그 블록을 영영 못 만짐. → 종료 시 "잡았던 블록 id"를 실어 명시 해제 + 본인 소유만 원자적 `remove(key,value)`.
- **removeCandidate TOCTOU**: "블록 있는지 검사 → 후보 삭제" 사이에 동시 배치가 끼면 FK(RESTRICT) 예외. → 삭제를 `DataIntegrityViolationException` 캐치로 감싸 동일 메시지(409)로 변환.

### ④ 고민·의사결정
- **자동 병합(CRDT/OT)을 왜 안 했나**: 블록을 op/CRDT로 재모델링하고 op 로그 영속·재동기화까지 가는 수 주 규모 전환인데, 우리 규모엔 과하다. 게다가 **CRDT/OT도 의미 충돌은 못 없앤다** — A는 9시, B는 14시로 같은 블록을 옮기면 둘 중 하나로 수렴할 뿐 "사용자가 원한 결과"를 보장하진 않는다(정합성만 보장). 그래서 "충돌 시 거부 → 재조회·재시도"가 더 단순·정직하다고 판단.
- **백로그를 기계적으로 구현하지 않음(중요)**: 1차 분석에서 "presence in-memory의 `computeIfAbsent` 복합 연산 비원자성"을 위험으로 적었지만, 재검증 결과 **`ConcurrentHashMap.computeIfAbsent`는 원자적**이라 오진이었다. 없는 버그를 막겠다고 커서 이동마다 도는 핫 패스에 락을 넣으면 성능만 깎인다 → 의도적으로 두고 분석을 정정했다. "백로그 항목마다 실제 위험인가·막는 비용이 합리적인가를 재검증"한 사례.
- **막을 가치 기준의 일관성**: "영속 데이터 유실 = 지금 막는다 / 표시·성능 문제 = 나중"으로 우선순위를 일관되게 잡았다. 그래서 같은 블록 충돌·시간 겹침은 하드 차단했고, broadcast 부하·깜빡임은 백로그로 미뤘다.
- **다인원 확장 인식**: 데이터 정합성은 인원이 늘어도 (낙관적 락+grab+권한세대로) 지켜지지만, 먼저 무너지는 건 **성능·UX**(409 거부율↑, broadcast ~O(n²), 재계산 중복)임을 명시했다.

## 3. 실시간 커서 트래킹 + 상대 좌표 동기화

### ① 왜 "상대 좌표 환산"이 필요한가
- **문제**: 처음엔 커서를 **절대 픽셀(clientX/Y)** 로 주고받았다. 그러나
  - 사용자마다 **모니터·창 크기**가 다르고,
  - 같은 시간표라도 **세로 스크롤 위치**가 다르며(A는 1~8시, B는 9~15시 표시),
  - 지도 패널 폭·열림 여부도 다르다.
- 절대좌표를 그대로 공유하면 "5시 성심당 블록을 가리킨다"가 상대 화면에선 엉뚱한 곳에 찍힌다.
- **목표**: 커서를 **"어느 영역(zone) + 그 zone의 의미 좌표"** 로 송수신해, 창·스크롤·폭이 달라도 모두가 **"같은 의미의 지점"** 을 가리키게 한다. (픽셀 단위 완벽 일치는 비목표 — **의미 단위 정확**이 목표)

### ② 어떻게 구현했나
- **좌표 모델**(전송 payload)
  - `zone`: `timetable` | `map` | `other`
  - timetable: `dayIndex`(어느 day 컬럼) + `colRatioX`(컬럼 내 가로 0~1) + `contentY`(시간축 콘텐츠 px)
  - map: `mapRatioX/mapRatioY`(컨테이너 내 비율 — lat/lng 정밀변환은 추후)
  - grab(드래그): `grabRatioX`(블록 폭 대비 잡은 가로) + `grabOffsetMin`(블록 top으로부터 분)
- **기준 DOM**: 세로는 `.day-cols`(시간 원점, 0~1440px 고정), 가로는 개별 `.day-col`. 두 요소의 `getBoundingClientRect()`가 **스크롤을 이미 반영**하므로 별도 scrollTop 보정이 필요 없다.
- **송신**(`buildPointerPayload`): 마우스 이벤트 → zone 판정 → `contentY = clientY - dayCols.top`, `colRatioX`, `dayIndex` 등으로 정규화.
- **수신**(`cursorStyle`/`ghostStyle`/`dropPreviewStyle`): payload를 **자기 DOM 기준**으로 역환산. 예) `top = 내 dayCols.top + contentY`, `left = 내 day-col.left + colRatioX × width`. 창·스크롤이 달라도 "같은 의미 위치"에 찍힘.
- **로직 분리**: 좌표 송수신을 `useCollabCursor.js` composable로 추출(2,000줄+ God Component `ScheduleBoard.vue`의 부담 분리).
- **표시 범위 제한**(두 종류, 서로 다른 문제)
  - *송신자 위치 기준*: 시간표 격자(`.day-cols`) 밖(헤더·시간축 거터·사이드바)은 zone='other'로 보내 숨김. 보드 밖으로 마우스가 나가면 `mouseleave`로 즉시 숨김.
  - *수신자 스크롤 기준*: 보내는 사람의 시간이 **받는 사람의 시야 밖**이면(스크롤 차이) 계산된 y가 시간표 위로 삐져나가 헤더에 그려진다 → 현재 보이는 격자 뷰포트(`gridViewport`) 밖이면 숨김.
- **presence 생명주기**
  - 화면에 있으면(idle여도) `keepalive` **heartbeat**(좌표/zone 미변경, lastSeen만 갱신)로 커서·아바타 유지.
  - 다른 창/탭으로 포커스를 잃으면(`blur`/`visibilitychange`) `leave` 전송 → presence에서 제거(아바타 원형까지 사라짐). 복귀 시 재등록.
  - 창 종료는 WS `onDisconnect`가 처리. 응답 없는 세션은 stale evict(기본 5초).
- **부드러움(적응형)**: 전송·보간 값을 부하에 따라 자동 조절.
  - 송신 throttle: AIMD — 움직임이 전송보다 빠르면(혼잡) 간격↑(back off), 한가하면 간격↓.
  - 수신 보간: CSS transition(`--collab-smooth`)을 **최근 수신 간격(dt)에 맞춤** → 한가하면 부드럽게(상한까지), 드래그로 촘촘하면 자동 스냅. transition이 항상 dt 이하라 **백로그가 구조적으로 불가능**.
  - 시연/실서비스 전환을 위해 throttle·smooth·keepalive 값을 `.env`(`VITE_COLLAB_*`)·`config/collab.js`로 분리.

### ③ 어려움
- **세로축 보정**: day-col 콘텐츠 높이가 모두 동일(1440px)이라 `contentY`(콘텐츠 좌표)로 일반화하면 스크롤 무관해진다는 걸 잡는 게 핵심이었다. 처음엔 일반 커서가 이 보정을 전혀 안 해 어긋났다.
- **드래그 후 ghost가 한참 기어감**: `onDragOver`가 매 이벤트마다 throttle 없이 커서를 보내 **백로그**가 쌓였다. 손을 떼도 큐가 천천히 소진되며 ghost가 늦게 따라옴. → 전송 throttle + (transition ≤ 수신 간격) 적응형으로 백로그 자체를 없앰.
- **idle 시 사라짐 vs 무관 영역 노출**: "가만히 있어도 유지"와 "엉뚱한 곳에선 숨김"이 상충하는 듯 보였지만, **송신자 위치 기준 숨김**과 **포커스 기준 유지**는 조건이 달라 양립함을 정리.
- **스크롤 차이로 헤더 침범**: 위에서 설명한 수신자-스크롤 문제. 커서가 `position:fixed`로 body에 teleport돼 클리핑이 없어 GNB/plan 헤더까지 올라가 그려졌다 → 가시 격자 범위 클램프로 해결.

### ④ 고민·의사결정
- **정확도 목표를 "의미 단위"로 못박음**: 픽셀 완벽 일치를 좇지 않고 "같은 day·시간·블록을 가리키면 OK"로 정의 → 구현 난이도와 효과의 균형.
- **지도는 비율 근사로 합의**: 지도 커서까지 lat/lng로 정밀 변환하면 줌·팬이 달라도 일치하지만, Naver projection 검증 비용이 커서 **비율 근사**로 두고 정밀화는 백로그(`improvements.md`)에 등재. 현재는 지도 zone 커서를 임시 숨김.
- **커서 가시화 = 동시성의 인적 안전장치**: 물리적 락은 아니지만 "지금 누가 어디를 만지는지"를 보여줘 **물리 충돌 빈도 자체를 낮추는 1차 완충**으로 봤다. 그래서 "다른 블록을 같은 슬롯에 놓는" 류의 드문 충돌은 하드 보장 대신 커서 인지 + 서버 정규화에 맡기는 판단이 가능했다(2장과 연결).
- **고정 튜닝 → 적응형**: 처음엔 throttle/smooth를 고정값으로 너프했는데, "리소스 여유 있으면 부드럽게, 쌓이면 즉시 낮춤"을 자동화하기 위해 AIMD·간격연동 보간으로 전환. 수동 튜닝 없이 부하에 반응.

## 4. 예상 질의응답 (요약)

- **왜 폴링/SSE 대신 WebSocket?** 커서 전송이 잦은 양방향 통신이라. STOMP로 토픽 pub/sub·채널 분리, SockJS로 폴백.
- **왜 비관적 락이 아니라 낙관적 락?** 충돌이 드문 환경. 비관적은 평소 손해, 낙관적은 평소 0·경합 때만 거부.
- **version이 다르면 무조건 충돌? transit 재계산은?** `version`은 같은 row에만 작용 + transit는 version을 안 건드리는 별도 UPDATE라 오탐 없음.
- **시간 겹침은 왜 version으로 못 막나?** cross-row 제약이라 단일 row version 밖. `trip` 행 `FOR UPDATE`로 직렬화해 해결.
- **충돌 나면 사용자 경험은?** 자동 병합 안 함. 진 쪽은 "먼저 수정됨" 안내 + 최신 재조회 후 재시도.
- **CRDT/OT는 왜 안 썼나?** 수 주 규모 전환 + 의미 충돌은 CRDT도 못 없앰. 규모 대비 ROI 낮음.
- **절대좌표 대신 왜 상대좌표?** 창 크기·스크롤·패널 폭이 달라 절대픽셀은 엉뚱한 곳에 찍힘. zone+의미좌표로 "같은 의미 지점" 보장.
- **스크롤이 다른데 커서가 헤더에 뜨던 문제는?** 받는 사람 시야(격자 뷰포트) 밖이면 숨김 처리.
- **드래그 후 커서가 늦게 따라오던 랙은?** 무throttle 전송 백로그가 원인. 적응형 throttle + (transition ≤ 수신간격) 보간으로 해결.
- **인원이 많아지면?** 정합성은 유지, 성능·UX(거부율, broadcast O(n²), 재계산 중복)가 먼저 병목. 보강 1순위는 broadcast 효율·재계산 디바운스.
