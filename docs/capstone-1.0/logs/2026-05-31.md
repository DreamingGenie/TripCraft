### `@RequestParam` name 속성 누락으로 인한 관광지 조회 500 오류 수정

---

#### 현상

`GET /api/attractions` 요청 시 500 에러 반환. 서버 로그:

```
java.lang.IllegalArgumentException: Name for argument of type [java.lang.String] not specified,
and parameter name information not available via reflection.
Ensure that the compiler uses the '-parameters' flag.
```

---

#### 원인

Spring MVC는 `@RequestParam`에 `name` 속성이 없으면 **Java 리플렉션으로 파라미터 이름을 자동으로 읽는다.**  
이를 위해 컴파일 시 `-parameters` 플래그가 필요하다.

`build.gradle.kts`에는 해당 플래그가 이미 설정되어 있었다:

```kotlin
// build.gradle.kts
tasks.withType<JavaCompile> {
    options.compilerArgs.add("-parameters")
}
```

그러나 **IDE(IntelliJ 등)에서 자체 컴파일러로 직접 실행하면 Gradle 설정이 적용되지 않아** 파라미터 이름 정보가 바이트코드에 포함되지 않는다.  
결과적으로 Spring이 파라미터 이름을 인식하지 못해 `IllegalArgumentException`이 발생했다.

> **참고**: 2026-05-28 일지에서 `build.gradle.kts`에 `-parameters` 플래그를 추가한 것은 Gradle 빌드(`./gradlew bootRun`) 기준으로는 효과가 있으나, IDE 직접 실행 시에는 IDE의 컴파일러 설정을 별도로 맞춰줘야 한다. 이번 수정은 IDE 환경에 관계없이 항상 동작하도록 코드 자체를 수정했다.

---

#### 해결 방법

`@RequestParam` 어노테이션 모두에 `name` 속성을 명시적으로 추가.  
이렇게 하면 리플렉션 없이 어노테이션 값으로 직접 파라미터 이름을 확인하므로 **컴파일 방식에 무관하게 동작**한다.

#### 향후 코드 작성 규칙

> **`@RequestParam`, `@RequestHeader` 등 이름 기반 어노테이션에는 항상 `name` 속성을 명시한다.**  
> IDE 종류나 빌드 방식에 관계없이 동일하게 동작하며, 코드의 의도도 명확해진다.

```java
// ❌ 지양
@RequestParam String keyword

// ✅ 권장
@RequestParam(name = "keyword") String keyword
```

### TourAPI 관광지 동기화 Spring Batch 적용

---

#### 배경 및 목적

기존 방식은 `POST /api/admin/attractions/sync` 엔드포인트를 수동으로 호출해야만 TourAPI 데이터를 갱신할 수 있었다.
이 방식은 아래 문제를 가지고 있었다.

- 수동 실행에 의존 — 관광지 데이터 최신화가 사람 손에 달려 있음
- 실패 시 처음부터 재실행 — 수만 건 처리 중 중단되면 전체를 다시 돌려야 함
- 진행 상황 추적 불가 — 몇 건이 처리됐는지, 어디서 실패했는지 알 수 없음
- 대용량 데이터를 한 번에 메모리에 올려 처리 — 안정성 취약

Spring Batch 적용으로 위 문제를 모두 해결했다.

---

#### 우리 프로젝트에서 얻는 효과

- **자동 실행**: 매일 새벽 2시(KST)에 스케줄러가 Job을 자동 실행. 수동 개입 불필요
- **청크 단위 안전 처리**: 100건씩 읽고 → 변환하고 → DB에 저장. 실패해도 해당 100건만 롤백
- **실행 이력 자동 저장**: `BATCH_JOB_EXECUTION` 등 메타테이블에 실행 시각, 처리 건수, 성공/실패 여부가 자동 기록
- **변환 실패 아이템 자동 skip**: Processor에서 변환 실패한 관광지는 null 반환으로 조용히 건너뜀. 일부 불량 데이터가 전체 배치를 중단시키지 않음
- **기존 수동 트리거 유지**: 기존 `POST /api/admin/attractions/sync` 엔드포인트는 그대로 사용 가능. 배치와 수동 실행이 공존

---

#### 배치 방식 선택 이유

| 결정 항목 | 선택 | 이유 |
|----------|------|------|
| 동기화 전략 | 전체 재수집 (Full Sync) | TourAPI의 `modifiedtime` 필터 신뢰도가 검증되지 않아 초기엔 전체 수집으로 안전하게 시작. 추후 증분으로 전환 가능 |
| Step 구성 | 청크 기반 단일 Step | 지역별 다중 Step은 Job 설정이 복잡해짐. 단일 Step + Reader 내부 순회로 구조를 단순하게 유지하면서 청크 처리의 이점을 모두 취함 |
| 메타테이블 위치 | 동일 DB (`trip_craft`) | 프로젝트 규모에서 DB 분리는 오버엔지니어링. 재시작 기능을 살리는 게 더 중요 |
| 실행 주기 | 매일 새벽 2시 | 서비스 트래픽이 가장 낮은 시간대. TourAPI 쿼터 소비도 분산 |

---

#### 실제 적용 내용

**신규 생성 파일 (6개)**

| 파일 | 경로 | 역할 |
|------|------|------|
| `TourApiItemReader.java` | `attraction/batch/` | 17개 지역 × 6개 콘텐츠 타입 = 102개 조합을 페이지 순회하며 TourApiItem을 1건씩 반환 |
| `TourApiItemProcessor.java` | `attraction/batch/` | TourApiItem → Attraction 변환. 실패 시 null 반환으로 해당 아이템 skip |
| `TourApiItemWriter.java` | `attraction/batch/` | 청크(100건) 단위로 `AttractionMapper.insertAll()` 호출 — ON DUPLICATE KEY UPDATE |
| `TourApiSyncJobConfig.java` | `attraction/batch/` | Job · Step · Bean(Reader/Processor/Writer) 등록. 청크 크기 100 |
| `TourApiSyncScheduler.java` | `attraction/batch/` | `@Scheduled(cron = "0 0 2 * * *")` — 매일 새벽 2시 Job 자동 실행 |
| `SchedulingConfig.java` | `global/config/` | `@EnableScheduling` 활성화 |

**수정 파일 (2개)**

`build.gradle.kts` — `spring-boot-starter-batch` 의존성 추가

```kotlin
implementation("org.springframework.boot:spring-boot-starter-batch")

```

`application.yml` — Batch 메타테이블 자동 생성 및 앱 시작 시 Job 자동 실행 방지

```yaml

  batch:
    jdbc:
      initialize-schema: always  # BATCH_JOB_INSTANCE 등 메타테이블 자동 생성
    job:
      enabled: false             # 앱 시작 시 Job 자동 실행 방지
```

#### 데이터 흐름

```
[매일 새벽 2시]
TourApiSyncScheduler
  → JobLauncher.run(tourApiSyncJob, {runDate: 오늘날짜})
      → tourApiSyncStep  (100건 청크 반복)
            read()   : TourApiItemReader  → TourApiItem 1건 반환
            process(): TourApiItemProcessor → Attraction으로 변환 (실패 시 skip)
            write()  : TourApiItemWriter  → attraction 테이블 UPSERT 100건
      → BATCH_JOB_EXECUTION 테이블에 SUCCESS/FAILED 자동 기록
```

#### @StepScope 적용 이유

Reader · Processor · Writer 모두 `@StepScope`로 등록했다.
Step이 시작될 때 인스턴스가 새로 생성되므로 Reader의 내부 상태(페이지 번호, 버퍼)가 매 실행마다 초기화된다.
`@StepScope` 없이 싱글톤으로 두면 두 번째 실행부터 Reader가 이미 소진된 상태로 시작해 아무것도 처리하지 않는 버그가 발생한다.

#### JobParameters에 runDate를 넣는 이유

Spring Batch는 동일한 JobParameters로 이미 성공한 Job의 재실행을 막는다.
`runDate=오늘날짜`를 매일 바꿔 전달함으로써 날마다 새 Job 인스턴스로 실행할 수 있다.
