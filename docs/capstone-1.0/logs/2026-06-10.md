# 작업 일지 — 2026-06-10

## 완료

### Transit 이동수단 계산 로직 전면 개선

---

#### 변경 배경

기존 구조의 두 가지 문제:
1. **블록 드롭 시** 대중교통 1개 모드만 계산하고, NONE 결과가 나오면 pill에 "경로 없음"으로 방치
2. **이동수단 모달** 열릴 때 버스·자동차·도보 3개 + 상세 경로 + 자동차 옵션 4개 = 총 5개 API를 한꺼번에 병렬 호출

---

#### 1. 드롭 시 fallback 로직 (백엔드)

`recalculateTransitForDate`에서 trip 기본 모드(PUBLIC_TRANSIT)로 계산했을 때 NONE이 반환되면
자동차로 fallback해 pill을 채운다.

추가로, 블록에 저장된 `transitMode`가 `BUS`, `SUBWAY` 같은 ODsay 결과 모드일 때
`getTransitTime`의 캐시 키(`requestMode`)는 `PUBLIC_TRANSIT/DRIVING/WALKING` 세 가지만 유효하므로
저장된 값을 올바른 요청 모드로 매핑하도록 수정했다.

| savedMode | effectiveMode |
|---|---|
| `DRIVING` | `DRIVING` |
| `WALKING` | `WALKING` |
| `BUS`, `SUBWAY`, `BUS,SUBWAY` 등 | `PUBLIC_TRANSIT` |
| `null`, `NONE` | trip 기본 모드 |

이 매핑이 없으면 `mode=BUS`로 캐시 조회 시 항상 miss → ODsay 재호출 → NONE 반환 →
`durationMinutes=null`이 저장되어 pill이 "분 · 버스"처럼 비정상 표시되는 버그가 있었다.

---

#### 2. 이동수단 모달 lazy 로딩 (프론트엔드)

**이전**: 모달 열릴 때 세 수단 + 버스 상세 + 자동차 옵션 4개 = 5개 병렬 호출

**이후**: 현재 탭 데이터만 조회, 탭 클릭 시점에 해당 탭 데이터 조회

```
pill 클릭
  → 현재 반영된 탭(예: PUBLIC_TRANSIT)만 summary + 상세 경로 조회

탭 클릭
  → 해당 탭 데이터만 lazy 조회 (이미 로드됐으면 skip)
```

---

#### 3. 자동차 옵션 lazy 로딩

기존에는 자동차 탭을 열면 추천·최단시간·무료도로·최소거리 4개를 한꺼번에 호출.
이제는 최초 진입 시 추천(index 0)만 조회하고, 각 탭 클릭 시점에 개별 호출한다.

**백엔드**: `getDrivingOption(optionIndex)` 단건 메서드 추가.
기존 `getDrivingOptions`(4개 일괄)는 유지하고, `/driving-options?optionIndex=0` 파라미터 추가 시
단건 조회로 분기.

**프론트엔드**: `pillDrivingOptions[key]`를 4-슬롯 배열(`[undefined, ...]`)로 관리.
슬롯별 로딩 스피너와 시간 표시를 독립적으로 처리.

### ODsay loadLane으로 버스·지하철 실제 노선 폴리라인 표시

---

#### 변경 배경

지도에서 대중교통 경로를 직선으로만 표시하고 있었다.
ODsay `searchPubTransPathT` 응답의 `subPath`에서 시작·끝 좌표만 연결하는 방식이었기 때문이다.
ODsay는 `loadLane` API로 버스·지하철 노선의 실제 도로/선로 폴리라인을 제공한다.

---

#### 구현 내용

**ODsay loadLane API 확인 (직접 호출 검증)**

`mapObject` 파라미터의 올바른 형식을 다양하게 시도해 직접 확인했다.

- 잘못된 형식: `@{idx}:{trafficType}:{routeID}:{startID}:{endID}` (subPath lane 정보로 직접 조합)
- **올바른 형식**: `"0:0@" + info.mapObj` — `searchPubTransPathT` 응답의 `info.mapObj` 값을 그대로 사용

`info.mapObj`는 이미 경로 탐색 결과에 포함되어 있어 추가 파싱 없이 바로 사용 가능하며,
경로 전체의 모든 교통 구간을 단일 API 호출로 처리할 수 있다.

응답 구조 확인:
```json
{
  "result": {
    "lane": [
      {
        "section": [
          { "graphPos": [{ "x": 127.026, "y": 37.500 }, ...] }
        ]
      }
    ]
  }
}
```

**3단계 fallback 구조**

```
1순위: ODsay loadLane (DB 캐시 우선, 없으면 API 호출 후 저장)
2순위: passStopList 정류장 좌표 (정류장을 이은 꺾인 선)
3순위: 구간 시작·끝 좌표 (기존 방식)
```

**lane_polyline 테이블 (DB 캐시)**

노선 형상은 자주 변경되지 않으므로 영구 캐시로 관리한다.
초최 조회 시에만 API를 호출하고 이후에는 DB에서 서빙한다.

```sql
CREATE TABLE lane_polyline (
    id             BIGINT       NOT NULL AUTO_INCREMENT,
    map_object_key VARCHAR(250) NOT NULL COMMENT 'ODsay info.mapObj 값 (캐시 키)',
    route_coords   MEDIUMTEXT   NULL     COMMENT '파싱된 좌표 JSON [[lng,lat],...]',
    raw_response   MEDIUMTEXT   NULL     COMMENT 'ODsay loadLane 원본 응답 JSON',
    created_at     DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (id),
    UNIQUE KEY uk_map_object_key (map_object_key)
);
```

`raw_response`에 원본 JSON을 함께 저장해 파싱 로직 변경 시 재가공이 가능하도록 했다.
