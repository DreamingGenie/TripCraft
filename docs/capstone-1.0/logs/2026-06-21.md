## 5. 실무적으로 고민한 포인트

### 5-1. RESTRICT FK 삭제 순서

`trip_block.candidate_id` FK가 `ON DELETE RESTRICT`로 설정되어 있다.

`trip`을 먼저 지우면 MySQL이 CASCADE로 `trip_candidate`를 삭제하려 하지만,
`trip_block`이 `trip_candidate`를 참조하고 있어 RESTRICT 위반으로 실패한다.

**반드시 `trip_block → trip_candidate → trip` 순서로 삭제해야 한다.**

이를 위해 세 Mapper에 `deleteByMemberId`를 추가했다.

```sql
-- 1단계: trip_block
DELETE tb FROM trip_block tb
JOIN trip_candidate tc ON tc.id = tb.candidate_id
JOIN trip t ON t.id = tc.trip_id
WHERE t.member_id = #{memberId}

-- 2단계: trip_candidate
DELETE tc FROM trip_candidate tc
JOIN trip t ON t.id = tc.trip_id
WHERE t.member_id = #{memberId}

-- 3단계: trip
DELETE FROM trip WHERE member_id = #{memberId}
```

### 5-2. 파일 삭제를 트랜잭션 커밋 이후로 분리

파일 삭제를 트랜잭션 안에서 먼저 실행하면 아래 시나리오에서 불일치가 발생한다.

```
[잘못된 순서]
파일 삭제 → DB 작업 → 트랜잭션 커밋
           ↑
           여기서 DB 작업 실패 시 롤백 → attach 레코드는 복원,
           하지만 파일은 이미 삭제된 상태 → 불일치
```

기존 `PostServiceImpl.deletePost`도 동일한 이유로 `@TransactionalEventListener(AFTER_COMMIT)` 패턴을 사용한다.
회원 탈퇴에서도 `TransactionSynchronizationManager.registerSynchronization`을 활용해 커밋 이후에 파일을 삭제하도록 구현했다.

```
[올바른 순서]
DB 작업 → 트랜잭션 커밋 → afterCommit 훅 → 파일 삭제
                        ↑
                        커밋이 성공해야 파일 삭제 시작
```

커밋 이후 파일 삭제가 실패하면 고아 파일이 남지만 DB 정합성은 이미 보장된 상태다.
실운영이라면 별도 모니터링이나 정리 배치가 필요한 지점이나, 캡스톤 규모에서는 허용 가능한 엣지 케이스로 판단했다.

### 5-3. @Transactional 위치 — 컨트롤러가 아닌 서비스 계층

초안에서는 `MemberController`에 직접 `@Transactional`을 붙였으나 리팩토링에서 수정했다.

Spring AOP 프록시 구조상 컨트롤러 public 메서드에 붙여도 동작은 하지만,
트랜잭션 관리는 서비스 계층의 책임이다. 컨트롤러는 요청을 받아 서비스에 위임하는 역할에만 집중해야 한다.

`MemberService` + `MemberServiceImpl`을 추가하고 비즈니스 로직과 `@Transactional`을 서비스로 이동했다.

### 5-4. post_draft 이미지 정리

탈퇴 시 프로필 이미지만 삭제하면 `target='post_draft'` 레코드가 남는다.

`post_draft`는 글 작성 중 에디터에서 이미지를 업로드할 때 생성되며, `target_id`가 `memberId`로 설정된다.
회원이 사라지면 이 파일들은 `OrphanImageCleanupScheduler`의 24시간 TTL 대상이 되긴 하지만,
탈퇴 시점에 즉시 정리하는 것이 더 명확하다.

`FILE_TARGETS = List.of("profile", "post_draft")`로 두 target을 한 번에 처리하도록 구현했다.

`target='post'` 이미지는 삭제하지 않는다. 해당 이미지는 게시글에 첨부된 것이고,
탈퇴 후에도 게시글은 "탈퇴한 사용자" 명의로 보존되므로 이미지도 유지해야 한다.

### 5-5. 비밀번호 검증 방식 — 존재 여부 비노출

```java
// 초안
Member member = memberMapper.findById(memberId).orElseThrow(() -> new ResponseStatusException(UNAUTHORIZED));
if (!passwordEncoder.matches(request.getPassword(), member.getPassword())) {
    throw new ResponseStatusException(BAD_REQUEST, "비밀번호가 올바르지 않습니다.");
}

// 최종
memberMapper.findById(memberId)
        .filter(m -> passwordEncoder.matches(password, m.getPassword()))
        .orElseThrow(() -> new ResponseStatusException(BAD_REQUEST, "비밀번호가 올바르지 않습니다."));
```

초안은 회원을 찾지 못할 때와 비밀번호 불일치일 때 다른 HTTP 상태와 메시지를 반환한다.
이는 공격자가 특정 계정의 존재 여부를 확인하는 데 악용될 수 있다.

최종 코드는 두 경우 모두 400 + 동일 메시지로 처리한다.
토큰 인증이 이미 통과된 시점(`@AuthenticationPrincipal`으로 memberId 확보)이므로
회원이 없는 경우는 사실상 발생하지 않지만, 방어적으로 동일 응답을 유지했다.

### 5-6. 캡스톤 규모라 생략한 것

| 항목 | 생략 이유 |

## 6. 프론트엔드 UX

`/mypage/profile` 페이지 맨 하단에 탈퇴 섹션을 추가했다.

```
[탈퇴하기] 버튼 클릭
    ↓
인라인 확인 폼 표시 (비밀번호 입력 + 취소/탈퇴확인 버튼)
    ↓
비밀번호 입력 후 [탈퇴 확인]
    ↓
DELETE /api/members/me 호출
    ↓
성공 → auth.logout() → 홈(/)으로 리다이렉트
실패 → 에러 메시지 인라인 표시
```

`window.confirm()` 대신 인라인 확인 플로우를 사용했다.
브라우저 기본 다이얼로그는 스타일이 통일되지 않고 UX가 투박하다는 이유로 기존 `MyPostsView`에서도 개선 대상으로 기록되어 있었다.
