## 슬라이드 27 — 왜 무료 API 조합인가
**도입(또박또박):** "'출발지에서 목적지까지 몇 분'을 구하는 가장 쉬운 길은 Kakao Mobility 같은 유료 단일 API입니다. 저희는 비용을 피해 무료 조합을 택했고, 그 대가가 이 장의 두 축입니다."

- 유료 단일 = 응답 그대로 쓰면 끝, 대신 호출당 과금. 무료 조합(ODsay·T Map·Naver) = 비용 0, 대신 기술 부담을 우리가 떠안음.
- 대가는 정확히 둘: ① 응답을 직접 **합성·파싱**(다음 두 장) ② 호출 제한을 **캐시로 방어**(그 다음 장).
- 어려움의 본질 셋으로 못 박기 — **파싱**(깊게 중첩된 ODsay JSON) · **조합**(도시간 경로 접근구간 보강 + 노선 폴리라인 별도 호출) · **호출 제한**(같은 좌표 반복 호출 시 빈 응답).

**전환:** "그래서 경로 하나를 만드는 데 API 4종이 얽힙니다 — 오케스트레이션부터 보시죠."

## 슬라이드 28 — 경로 API 오케스트레이션 (다이어그램)
**도입:** "경로 하나에 외부 API 4종이 관여합니다. 프론트가 '모드'를 넘기면 백엔드가 해당 API를 호출·합성해 좌표와 요약을 만들고, 프론트는 네이버 지도에 그리기만 합니다."

- 진입점은 백엔드 `TransitServiceImpl.getTransitTime()` (`/api/transit`). 프론트가 보낸 `mode`로 분기:
  - `DRIVING`/`WALKING` → `TMapClient` (자동차 예측 API / 보행자 API)
  - `PUBLIC_TRANSIT` → `OdsayClient`의 `searchPubTransPathT` 호출 후 **합성 단계(enrich)**로
- **설계 의도**: 외부 API 호출·합성·캐싱을 전부 백엔드에 격리. 프론트는 "좌표 + 요약(분)"만 받아 렌더 → 키 노출 방지 + 캐시 공유 + 프론트 단순화.
- Naver는 길찾기가 아니라 **렌더 전용**임을 분명히.

**전환:** "그런데 대중교통, 특히 도시 간 경로는 응답을 그대로 못 씁니다. 여기서 '합성'이 필요합니다."

## 슬라이드 29 — 경로 합성 · 보강 (조립) ★기술 하이라이트
**도입:** "ODsay의 도시 간 경로는 '역에서 역까지'만 줍니다. KTX 구간만 받으면 소요시간이 비현실적으로 짧게 나오죠. 그래서 앞뒤 접근 구간을 직접 이어 붙입니다."

- **도시간 판별**: `enrichPublicTransit()`에서 ODsay `pathType`으로 구분 — `pathType < 11` 시내 경로(그대로), `>= 11` 도시간 → 보강 진입.
- **접근 구간 합성**: 출발지→출발역 = `localFrom`, 도착역→목적지 = `localTo`를 `OdsayClient.findLocalPath()`(SearchType=0)로 따로 호출.
- **핵심 공식**: `총분 = localFrom + interTime(도시간) + localTo`. 이 보강으로 "강남→부산역 KTX만 18분" 같은 거짓 소요시간이 교정됨.
- **폴백**: `findLocalPath` 실패 시 `haversineMinutes()`로 직선거리 추정(접근 구간 평균 30km/h, ODsay -98 '너무 가까움'은 5km/h). → 값을 못 구해도 빈칸이 아니라 합리적 추정으로 메움.
- **경로별 독립 계산 + 좌표키 캐시**: 출발역이 다른 후보(수서/서울/강남)는 각자 역 좌표로 따로 계산. 같은 역을 공유하는 경로가 많아 **좌표키로 중복 제거** → 20~40회 호출을 고유 역 수(1~3회)로 절감.

**전환:** "방금 '중복 제거'를 말했는데 — 무료 API의 진짜 적은 호출 제한입니다. 캐시 설계를 보시죠."

## 슬라이드 30 — 다층 캐싱 (DB 2 + in-memory 2) ★기술 하이라이트
**도입:** "무료 API는 같은 좌표를 짧은 시간에 반복 호출하면 빈 응답을 줍니다. 그래서 DB 2계층, 메모리 2계층 — 총 4겹 캐시를 설계했고, 관통하는 원칙이 하나 있습니다."

- **① DB `transit_cache`** — 키 `(from_attraction_id, to_attraction_id, departure_hour, request_mode)`. 모드·시간대별 행. **실패(NONE)도 저장**(`saveNoneCache`)해 같은 실패 재호출 방지.
- **② DB `lane_polyline`** — 노선 형상(`loadLane`) 영구 캐시. 키는 ODsay `mapObj` 기반(`"0:0@"+mapObj`). **결과 없음(null)도 저장**해 무의미한 재시도 차단.
- **③ in-memory `publicCoordCache`** — 커스텀 장소(by-coords) 경로용. 키 `"%.5f,%.5f>%.5f,%.5f"`(소수 5자리 좌표쌍). **성공만 저장**, 1000개 초과 시 clear.
- **④ in-memory `localPathCache`**(OdsayClient 내부) — 29장 접근구간 보강용. **같은 역쌍 중복 제거**, 2000개 상한, 성공·추정(-98)만 저장.
- **관통 원칙(가장 강조):** "**빈 응답은 저장하지 않는다.**" → 일시적 rate-limit이 영구 빈칸으로 굳지 않고 재시도됨. 결정적 결과(성공·없음 확정·추정)는 저장해 재호출 차단. **'무엇을 캐시하지 않느냐'가 설계 포인트.**

**전환:** "이렇게 백엔드가 만든 좌표를, 프론트는 지도에 정밀하게 그립니다."

## 슬라이드 31 — 지도 폴리라인 시각화
**도입:** "백엔드가 만든 좌표를 프론트(`ScheduleBoard.vue`의 `drawDayRoute`)가 네이버 지도에 그립니다. 그냥 선이 아니라 실제 노선처럼 보이게 했습니다."

- **좌표계 변환**: 저장은 `[lng, lat]`, Naver는 `LatLng(lat, lng)` → 그릴 때 swap.
- **구간별 색**: 지하철 호선 공식색(`subwayColorByName`: 1호선 #0052A4, 2호선 #00A84D…), 버스 종류색(`BUS_TYPE_COLORS`), KTX 검정(`RAIL_COLOR`), 도보 파란 점선. 색선 아래 **검정 casing**(zIndex 5, 굵기+3)을 깔아 가독성 확보.
- **오버레이**: 방향 화살표 ~30px 간격(`ARROW_SPACING_PX`), 역·정류장 마커, 노선 라벨은 **7단 오프셋으로 충돌 회피** + 리더선.
- **3단 폴백**: ① `drawPublicSegments`(실 노선 폴리라인) → ② API `routeCoords` → ③ 양끝 직선(`drawStraightFallback`). + `drawGeneration` 카운터로 빠른 재호출 시 **stale 렌더 방지**.
- (시연 이미지 가리키며) "왼쪽 대중교통 구간색 경로, 오른쪽 T Map 자동차 경로"로 마무리.

**전환:** "여기까지가 가장 공들인 경로 파트였고, 다음은 실시간 협업 동시성입니다." (→ 슬라이드 32~)

## 예상 Q&A (깊은 기술 대비)
- **Q. 유료 API가 더 쉬운데 왜 굳이?** → 비용도 있지만, 대중교통은 무료 ODsay가 노선·환승 데이터가 풍부. 트레이드오프를 '합성·캐싱' 기술로 흡수.
- **Q. 캐시 무효화/신선도는?** → DB 캐시는 `departure_hour`를 키에 포함해 시간대별 분리. 노선 형상은 사실상 불변이라 영구 캐시. 빈 응답 미저장이라 일시 오류는 자동 회복.
- **Q. 좌표 캐시 메모리 누수 위험?** → 상한(1000/2000) 초과 시 비우는 단순 정책. LRU는 아니지만 캐시 목적(반복 호출 방어)에 충분, 구현 단순성 우선.
- **Q. haversine 30km/h 근거?** → 접근 구간(도보+시내 대중교통 혼합)의 보수적 평균. 실패 시 '0분/거짓값'보다 합리적. -98(너무 가까움)은 도보로 보고 5km/h.
- **Q. 동시 호출/스레드 안전?** → 메모리 캐시는 `ConcurrentHashMap`. 렌더는 `drawGeneration` 세대 카운터로 늦게 도착한 응답을 폐기.
- **Q. ODsay `pathType < 11` 기준?** → ODsay 응답 규격상 시내(<11) vs 도시간(≥11) 구분. 도시간만 접근구간 보강 필요.
