## 0. 협업 편집 기능 구현

### 0-1. 전체 흐름

```
[프론트] ScheduleView 진입
  → collab.js: REST GET /api/trips/{id} (토큰 최신화)
  → SockJS /ws 연결 → STOMP CONNECT
  → JwtHandshakeInterceptor: access_token 쿠키 검증 → memberId 세션 저장
  → JwtChannelInterceptor: CONNECT 프레임에서 Principal 설정
  → SUBSCRIBE /topic/trip/{id}, /topic/trip/{id}/presence
  → 이후 모든 편집 이벤트는 WebSocket으로 수신
```

### 0-2. 인증 — 쿠키를 쓴 이유

SockJS 핸드셰이크는 HTTP 요청이므로 쿠키가 자동으로 전송된다.
`access_token` httpOnly 쿠키를 그대로 활용하면 별도 헤더 처리 없이 인증이 완성된다.

`JwtHandshakeInterceptor`가 HTTP 핸드셰이크 시점에 토큰을 읽어 `memberId`를 WebSocket 세션 attributes에 저장하고,
이후 STOMP 프레임마다 `JwtChannelInterceptor`가 그 값을 꺼내 Principal을 설정하거나 권한을 확인한다.

한 가지 엣지 케이스가 있다: 핸드셰이크 직전에 토큰이 만료될 수 있다.
이를 대비해 `_doConnect()` 시작 시 REST 요청을 먼저 보낸다. `http.js`의 인터셉터가 만료된 토큰을 자동 갱신하므로
핸드셰이크 쿠키는 항상 유효한 상태가 보장된다.

```js
async function _doConnect(tripId) {
  try { await tripApi.get(tripId) }  // 토큰 갱신 트리거
  catch { connected.value = false; return }
  stompClient = new Client({ ... })
  stompClient.activate()
}
```

### 0-3. TripRole — OWNER / EDITOR / VIEWER

서비스 레이어에 `resolveRole(tripId, memberId)` 헬퍼를 만들고,
이를 기반으로 세 가지 권한 단계를 구분한다.

```java
private void assertCanView(Long tripId, Long memberId)   // VIEWER 이상
private void assertCanEdit(Long tripId, Long memberId)   // EDITOR 이상
private void assertCanManage(Long tripId, Long memberId) // OWNER만
```

모든 Trip API 진입점에서 반드시 세 헬퍼 중 하나를 호출하도록 강제한다.
헬퍼 없이 직접 DB 접근하는 메서드는 만들지 않는다.

### 0-4. TripPresenceController — 실시간 커서·그랩 관리

협업자가 마우스를 움직이거나 블록을 잡으면
프론트가 `/app/trip/{id}/pointer`로 포인터 이벤트를 전송한다.

`TripPresenceController`가 이것을 받아 인메모리 `presenceMap`에 상태를 저장하고
`/topic/trip/{id}/presence`로 브로드캐스트한다.

```
tripId → sessionId → PresenceState(memberId, nickname, x, y, interaction, targetBlockId, lastSeen)
tripId → blockId → memberId  (grab 소유자)
```

**stale 처리**: `@Scheduled(fixedDelay = 2000)`로 2초마다 `lastSeen`이 5초 이상 지난 항목을 제거한다.
포인터 이벤트를 보내지 않는(마우스를 놓은) 사용자가 자동으로 presence 목록에서 빠진다.

**grab 충돌 방지**: 블록을 잡고 있는 사람이 누구인지 서버가 기억하고,
클라이언트는 `grabMap`을 보고 다른 사람이 잡은 블록을 드래그하지 못하도록 막는다.

```js
function isGrabbedByOther(blockId, myMemberId) {
  const owner = grabMap.value[blockId]
  return owner != null && owner !== myMemberId
}
```

### 0-5. 이벤트 브로드캐스트 — TripEvent

블록 추가·이동·삭제, 후보군 추가·제거, 이동시간 재계산 완료 등
일정 상태가 변하는 모든 서비스 메서드 끝에 `broadcast()`를 호출한다.

```java
broadcast(tripId, TripEvent.of("BLOCK_ADDED", memberId, nickname(memberId),
    Map.of("blockId", block.getId(), "candidateId", req.getCandidateId(), ...)));
```

프론트는 이벤트 타입별로 로컬 상태를 갱신한다.
`loadTrip()` 전체를 다시 호출하는 것보다 훨씬 빠르고, 다른 협업자의 작업이 즉시 반영된다.

### 0-6. CollaboratorPanel — 협업자 초대·관리 UI

OWNER만 볼 수 있는 초대 검색창과 협업자 목록을 `CollaboratorPanel.vue`로 분리했다.
닉네임·이메일 검색은 300ms debounce로 처리한다.

새 여행 일정을 만들거나 협업자가 삭제되어 403이 반환되면,
`ScheduleView`가 드롭다운 목록을 즉시 갱신하도록 콜백을 주입했다.

```js
// 일정 생성 시 새 항목을 목록에 즉시 추가
openScheduleModal({ onCreated: (newTrip) => myTrips.value.push(newTrip) })

// 403 수신 시 해당 일정을 목록에서 제거
catch (e) {
  if (e.status === 403) collaboratingTrips.value = collaboratingTrips.value.filter(t => t.id !== id)
}
```

## 1. `isReconnecting` 플래그 — 첫 연결과 재연결을 구분하는 이유

### 문제 발견 경위

기능을 테스트하던 중 일정 페이지 진입 시 `loadTrip()`이 두 번 호출되는 것을 콘솔에서 확인했다.
`onMounted`에서 한 번, STOMP `onConnect` 콜백에서 또 한 번이었다.

### 왜 이런 설계가 됐나

재연결 시 놓친 변경사항을 동기화하기 위해 `onConnect`에서 `loadTrip()`을 호출하는 구조로 설계했는데,
문제는 **최초 연결도 `onConnect`를 통과한다**는 점이다.
STOMP 클라이언트 입장에서 첫 연결과 재연결은 모두 `onConnect`라는 동일한 이벤트다.

### 해결 방식 — 외부 플래그

`isReconnecting`을 모듈 스코프 변수로 두고,
- `_doConnect()` 최초 호출 시: false (기본값 유지)
- `onDisconnect` 핸들러에서 재연결 예약 시: true로 전환
- `onConnect` 콜백에서 조건부 실행 후: false로 리셋

```js
// onDisconnect
isReconnecting = true
reconnectTimer = setTimeout(() => _doConnect(tripId), 3000)

// onConnect
if (isReconnecting && onReconnect) {
  try { await onReconnect() }
  catch (e) { console.warn('[collab] reconnect sync failed', e) }
}
isReconnecting = false
```

`disconnect()` 호출 시에도 `isReconnecting = false`를 리셋해야 한다.
탭 전환 등으로 명시적으로 연결을 끊었다가 재진입하는 경우,
`isReconnecting`이 true인 채로 다음 `connect()`가 호출되면
최초 연결인데도 `loadTrip()`이 실행된다.

## 2. `myRole` — 역할 판단을 클라이언트에서 서버로

### 기존 방식의 문제

이전에는 프론트엔드에서 "내가 OWNER인가"를 `myTrips` 목록에 해당 `tripId`가 있는지로 판단했다.

```js
// 기존 — 내 일정 목록에 있으면 오너
const activeTripIsOwner = computed(() =>
  myTrips.value.some(t => t.id === activeTripId.value)
)
```

이 방식은 두 가지 문제가 있다.

첫째, **협업으로 참여한 일정도 "내 일정"으로 표시될 수 있다.** `myTrips`와 `collaboratingTrips`를 구분해서 관리하면 회피 가능하지만, 그 분리 자체가 클라이언트 로직에 의존하므로 취약하다.

둘째, **역할 판단이 서버 없이 이루어진다.** 서버에서 EDITOR로 강등시켜도 클라이언트가 다음 전체 로드 전까지 OWNER처럼 행동할 수 있다.

### 해결 방식

`TripDetailResponse`에 `myRole` 필드를 추가하고, 서비스에서 `resolveRole()`로 채우도록 했다.

```java
String myRole = resolveRole(tripId, memberId).name();  // OWNER | EDITOR | VIEWER
```

프론트엔드에서는 단순히 비교한다.

```js
const activeTripIsOwner = computed(() =>
  activeTrip.value?.myRole === 'OWNER'
)
```

REST 응답에서 오는 값이므로 매 `loadTrip()` 호출마다 갱신된다.
서버가 권위를 가지고, 클라이언트는 그것을 표시할 뿐이다.

## 3. `placeBlock` 후보군 소유권 검증 — 누락된 보안 검사

### 발견된 취약점

블록을 일정에 배치하는 API(`POST /api/trips/{tripId}/blocks`)는
요청 바디로 `candidateId`를 받는다.
기존 코드는 해당 `candidate`가 실제로 이 `tripId`에 속하는지 확인하지 않았다.

이론상 다른 사람의 일정에 있는 `candidateId`를 넘기면
자신의 일정에 다른 사람의 후보군을 블록으로 생성할 수 있다.

### 왜 바로 에러가 나지 않았나

DB 스키마에서 `trip_block.candidate_id`의 FK는 `trip_candidate(id)`를 참조하지만
`trip_candidate.trip_id`는 `trip_block`이 속한 trip과 일치하지 않아도 FK 위반이 아니다.
즉, **외래키는 참조 무결성만 보장하고 도메인 무결성은 보장하지 않는다.**

### 수정 내용

```java
TripCandidate candidate = candidateMapper.findById(req.getCandidateId())
    .orElseThrow(() -> new ResponseStatusException(HttpStatus.NOT_FOUND, "후보군을 찾을 수 없습니다."));
if (!candidate.getTripId().equals(tripId))
    throw new ResponseStatusException(HttpStatus.FORBIDDEN, "다른 일정의 후보군입니다.");
```

애플리케이션 레이어에서 명시적으로 검증해야 하는 것들은 FK로 대체할 수 없다.
FK는 "존재하는 레코드를 참조"를 보장하지, "올바른 레코드를 참조"는 보장하지 않는다.

## 4. `JwtChannelInterceptor` — SEND 프레임 권한 검증과 캐시

### 기존 구조의 허점

기존 `JwtChannelInterceptor`는 `CONNECT`와 `SUBSCRIBE`만 처리하고
`SEND` 프레임은 별도 검증 없이 통과시켰다.

현재 `TripPresenceController`가 `@MessageMapping`에서 권한을 재확인하고 있어
실질적 취약점은 아니었지만, **새 WebSocket 핸들러를 추가할 때마다 권한 검증을 직접 작성해야 하는 구조**다.
인터셉터 레이어에서 한 번에 처리하는 것이 더 안전하다.

### SUBSCRIBE 캐시 추가 이유

같은 `tripId`에 대해 클라이언트는 두 채널을 구독한다.

```
/topic/trip/{tripId}          ← 일정 이벤트
/topic/trip/{tripId}/presence ← 참석자 현황
```

두 SUBSCRIBE 프레임이 거의 동시에 도착하므로 권한 DB 조회가 두 번 일어난다.
세션 어트리뷰트에 `tripAccess:{tripId} = true`를 캐시하면 두 번째 조회는 생략된다.

```java
String cacheKey = "tripAccess:" + tripId;
if (Boolean.TRUE.equals(accessor.getSessionAttributes().get(cacheKey))) return;
// ... DB 조회 ...
accessor.getSessionAttributes().put(cacheKey, true);
```

`SessionAttributes`는 WebSocket 세션 수명과 일치하므로 별도 TTL 관리가 필요 없다.
SEND 프레임도 같은 캐시 키를 공유하여 SUBSCRIBE 이후의 SEND는 DB 조회를 건너뛴다.

## 5. UI의 `alert()` → 인라인 확인 플로우로 교체

`CollaboratorPanel.vue`의 협업자 삭제 버튼이 `window.confirm()`을 사용하고 있었다.

브라우저 기본 다이얼로그는 스타일 일관성이 없고 UX가 투박하다.
더 중요한 문제는 **일부 브라우저 환경에서 `confirm()`이 항상 false를 반환**하거나 완전히 차단되는 경우가 있다.

인라인으로 "삭제? 확인 / 취소" UI를 구현했다.

```vue
<template v-if="confirmRemoveId === p.memberId">
  <button @click="remove(p.memberId)">확인</button>
  <button @click="confirmRemoveId = null">취소</button>
</template>
<button v-else @click="confirmRemoveId = p.memberId">삭제</button>
```

`confirmRemoveId` ref 하나로 어떤 항목이 "삭제 대기 중"인지 추적한다.
다른 항목 삭제 버튼을 누르면 자연스럽게 이전 확인 상태가 닫힌다.

## 6. rebase 과정에서 만난 팀원 변경사항

### 충돌 지점

팀원의 `master` 커밋이 `AttractionService`와 하드코딩된 `SIDO_NAME`/`TYPE_NAME` 맵을
`RegionService` + DB 기반 조회로 교체했다.

`feature/collab`에서는 `TripServiceImpl`에 `SimpMessagingTemplate`과
`MemberMapper`를 추가한 상태였으므로 `TripServiceImpl`에서 충돌이 발생했다.

해결 방향은 단순했다: **두 변경을 모두 살린다.**
팀원의 `RegionService` 의존성을 유지하면서 collab 브랜치의 `SimpMessagingTemplate`을 함께 주입한다.

```java
// before (collab branch)
private final AttractionService attractionService;
private final Map<Integer, String> SIDO_NAME = ...;

// after (rebase resolved)
private final RegionService regionService;  // 팀원 변경 수용
private final SimpMessagingTemplate messaging;  // collab 추가 유지
```

### 새로 생긴 `sido` 테이블 문제

rebase 후 서버를 재시작하니 `Table 'trip_craft.sido' doesn't exist` 500 에러가 발생했다.

팀원 커밋이 `sido` / `sigungu` 테이블을 새로 도입했는데,
마이그레이션 SQL은 `docs/02_design/migration_sido_v1.sql`에 있었지만
로컬 DB에 아직 적용되지 않은 상태였다.

코드는 머지됐는데 스키마가 뒤따라오지 않은 것이다.

이것이 마이그레이션 관리 도구(`Flyway`, `Liquibase`)가 필요한 이유다.
현재 프로젝트는 마이그레이션 파일을 docs에 수동으로 관리하고 있어,
rebase/머지 후 새 파일을 직접 확인하고 적용해야 한다는 인지 부담이 생긴다.

수동 마이그레이션 관리 체계에서 팀 작업 시 지켜야 할 원칙:
- 새 테이블·컬럼을 추가하는 커밋은 반드시 `docs/02_design/migration_*.sql`을 동반한다
- rebase/머지 후 `docs/02_design/`의 신규 파일을 확인하고 즉시 적용한다

## 7. WebSocket 연결 상태 표시 — 닷(·) 하나의 의미

협업 패널 버튼 옆에 초록/회색 점을 추가했다.

구현은 단순하다.

```js
// collab store
const connected = ref(false)
// onConnect → connected.value = true
// onDisconnect → connected.value = false
```

```vue
<!-- ScheduleView.vue -->
<span class="status-dot" :class="{ connected: collabStore.connected }"></span>
```

```css
.status-dot { background: var(--color-gray-400); }
.status-dot.connected { background: var(--color-success); }
```

단순해 보이지만 이 점이 있고 없고의 차이가 크다.

WebSocket은 끊어지면 사용자가 알 방법이 없다.
REST API처럼 요청-응답이 없으므로 에러 응답도 없다.
상태 표시 없이 그냥 메시지를 보내면 서버에 전달되지 않고 silently drop된다.

`connected` 상태를 시각적으로 드러냄으로써
"지금 실시간 연결이 살아있는지"를 사용자와 개발자 모두 즉시 알 수 있게 된다.
재연결 중(회색 점)인지, 정상 연결(초록 점)인지를 구분하는 것만으로도
디버깅 소요 시간이 크게 줄어든다.
