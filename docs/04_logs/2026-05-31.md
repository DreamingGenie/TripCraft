# 작업 일지 — 2026-05-31

## 완료

### `@RequestParam` name 속성 누락으로 인한 관광지 조회 500 오류 수정

---

#### 현상

`GET /api/attractions` 요청 시 500 에러 반환. 서버 로그:

```
java.lang.IllegalArgumentException: Name for argument of type [java.lang.String] not specified,
and parameter name information not available via reflection.
Ensure that the compiler uses the '-parameters' flag.
```

---

#### 원인

Spring MVC는 `@RequestParam`에 `name` 속성이 없으면 **Java 리플렉션으로 파라미터 이름을 자동으로 읽는다.**  
이를 위해 컴파일 시 `-parameters` 플래그가 필요하다.

`build.gradle.kts`에는 해당 플래그가 이미 설정되어 있었다:

```kotlin
// build.gradle.kts
tasks.withType<JavaCompile> {
    options.compilerArgs.add("-parameters")
}
```

그러나 **IDE(IntelliJ 등)에서 자체 컴파일러로 직접 실행하면 Gradle 설정이 적용되지 않아** 파라미터 이름 정보가 바이트코드에 포함되지 않는다.  
결과적으로 Spring이 파라미터 이름을 인식하지 못해 `IllegalArgumentException`이 발생했다.

> **참고**: 2026-05-28 일지에서 `build.gradle.kts`에 `-parameters` 플래그를 추가한 것은 Gradle 빌드(`./gradlew bootRun`) 기준으로는 효과가 있으나, IDE 직접 실행 시에는 IDE의 컴파일러 설정을 별도로 맞춰줘야 한다. 이번 수정은 IDE 환경에 관계없이 항상 동작하도록 코드 자체를 수정했다.

---

#### 해결 방법

`@RequestParam` 어노테이션 모두에 `name` 속성을 명시적으로 추가.  
이렇게 하면 리플렉션 없이 어노테이션 값으로 직접 파라미터 이름을 확인하므로 **컴파일 방식에 무관하게 동작**한다.

#### 향후 코드 작성 규칙

> **`@RequestParam`, `@RequestHeader` 등 이름 기반 어노테이션에는 항상 `name` 속성을 명시한다.**  
> IDE 종류나 빌드 방식에 관계없이 동일하게 동작하며, 코드의 의도도 명확해진다.

```java
// ❌ 지양
@RequestParam String keyword

// ✅ 권장
@RequestParam(name = "keyword") String keyword
```
