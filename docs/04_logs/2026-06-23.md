## 작업 개요

전날 구현한 협업 편집 기능 위에 **실시간 시각 피드백**(협업자 커서·드래그 유령 블록·drop-preview)을 입히고,
presence 동기화의 디테일을 다듬은 뒤, 팀원이 `master`에 머지한 여행 작업실(`/plan`) 구조에 맞춰
협업 코드를 `ScheduleBoard` 공용 컴포넌트로 이식했다.

오늘 작업의 큰 흐름은 세 덩어리다.
1. 협업자의 행동을 "보이게" 만들기 (커서, 유령 블록, 실루엣)
2. 그 시각화가 스크롤·드래그 오프셋에서도 정확히 맞도록 좌표 계산 다듬기
3. 변경된 컴포넌트 구조에 맞춰 이식

## 1. 협업자 커서·드래그 실루엣 실시간 시각화

### 무엇을 만들었나

다른 협업자가 지금 어디를 가리키고 무엇을 옮기는지를 화면에 실시간으로 그렸다.

- **커서**: SVG 화살표 + 닉네임 라벨이 협업자 마우스를 따라다닌다.
- **유령 블록(ghost block)**: 누군가 블록을 드래그하면, 그 블록의 복제본이 상대 커서 위치를 반투명으로 따라다닌다.
- **실루엣(grabbed-by-other)**: 원래 자리의 블록은 반투명 점선으로 바뀌고, 내가 잡을 수 없게 잠긴다.

### 사용한 것

- **`Teleport`**: 커서·유령 블록을 `body` 직속으로 렌더링한다. 시간표 컨테이너는 `overflow` 클리핑·`transform`·`z-index` 컨텍스트가 얽혀 있어, 그 안에서 `position: fixed`를 써도 의도대로 화면 절대 좌표에 붙지 않는다. `Teleport`로 DOM 트리 위치를 body로 끌어올려 클리핑·스택 컨텍스트 영향을 끊었다.
- **결정론적 색상 팔레트**: `collab.js`의 `colorMap`에서 협업자별 고유 색을 배정한다. 처음엔 입장 순서대로 색을 나눠줬는데, 그러면 누가 먼저 들어왔느냐에 따라 같은 사람이 매번 다른 색이 됐다. `memberId % PALETTE.length`로 바꿔서 **누가 보든 같은 사람은 항상 같은 색**이 되도록 했다.
- **포인터 전송**: `mousemove`와 `dragover`에서 `sendPointer`로 좌표·상태를 STOMP로 흘려보낸다. mousemove는 50ms throttle을 걸어 메시지 폭주를 막았다.

### 고민했던 포인트 — "그냥 커서"와 "잡은 상태"를 한 채널로

커서 이동과 블록 드래그는 둘 다 포인터 이벤트지만 의미가 다르다.
별도 메시지 타입으로 나눌까 하다가, `interaction` 필드(`''` vs `'grab'`) 하나로 구분하고
페이로드에 `targetBlockId`를 실어보내는 단일 채널로 합쳤다.
수신 측은 `interaction` 값만 보고 커서를 그릴지 유령 블록을 그릴지 분기한다.
메시지 종류를 늘리지 않아 서버 핸들러와 프론트 구독 로직이 단순해졌다.

## 2. presence 페이로드 확장 + keepalive heartbeat

### 문제 1 — 마우스를 멈추면 아바타가 사라진다

presence는 `@Scheduled`로 5초 이상 포인터 이벤트가 없는 세션을 stale로 보고 제거한다(전날 구현).
그런데 이 설계에는 부작용이 있었다: **마우스를 가만히 두면 5초 뒤 협업자 목록에서 사라진다.**
가만히 있는 것과 나간 것을 서버가 구분하지 못한다.

→ 프론트에서 4초마다 빈 포인터(keepalive heartbeat)를 보내도록 했다.
stale 기준(5초)보다 짧은 주기(4초)라야 "살아있지만 가만히 있는" 사용자가 목록에서 빠지지 않는다.
heartbeat 주기와 eviction 임계값의 관계(4 < 5)가 핵심이다.

### 문제 2 — drop-preview를 그리려면 좌표가 더 필요하다

유령 블록을 띄우는 것까지는 됐지만, "어느 칸에 떨어질지"를 미리 보여주려면
송신자의 스냅 위치를 받아야 했다. `PresenceState`에 다섯 필드를 추가했다.

```
snapDayIndex, snapTop   → 드롭될 Day 칸과 세로 위치
cursorRelY              → 스크롤 보정용 상대 Y
grabOffsetX, grabOffsetY → 블록의 어디를 집었는지
```

`broadcastPresence`를 `Map.of` → `HashMap`으로 바꾼 이유도 여기 있다.
`Map.of`는 null 값을 허용하지 않는데, 커서만 움직일 땐 `targetBlockId`가 null이라 직렬화에서 터졌다.

## 3. 드래그 ghost 위치 정확도 — 가장 까다로웠던 부분

오늘 시간을 가장 많이 쓴 작업. 유령 블록을 "띄우는" 건 쉬웠지만 **정확한 위치에 띄우는** 건 어려웠다.

### 어려움 1 — 집은 위치(grab offset)

블록의 가운데를 집든 모서리를 집든, 내 화면에서 드래그 이미지는 집은 지점을 기준으로 따라온다.
그런데 협업자 화면의 유령 블록은 커서 좌표에만 붙어서, 항상 블록 좌상단이 커서에 달라붙었다.

→ 드래그 시작 시 `grabOffsetX/Y`(블록 좌상단 기준 집은 지점)를 계산해 함께 전송하고,
수신 측이 `커서 - grabOffset`으로 유령 블록 좌상단을 역산하게 했다.
이걸 맞추니 "내가 보는 내 드래그"와 "남이 보는 내 드래그"의 손맛이 일치했다.

### 어려움 2 — 스크롤 불일치

가장 골치 아팠던 지점. 송신자와 수신자의 시간표 스크롤 위치가 다르면 유령 블록이 엉뚱한 곳에 떴다.
`day-col`의 `getBoundingClientRect()`는 스크롤된 뷰포트 기준이라, 양쪽 스크롤이 다르면 절대 좌표가 어긋난다.

→ 좌표 기준을 `wrapperEl`(스크롤 컨테이너)로 통일했다.
- 송신: `cursorRelY = senderScrollTop + clientY - wrapperTop` (스크롤 포함 절대 Y)
- 수신: `collabGhostTop = wrapperTop - receiverScroll + cursorRelY - grabOffsetY`

여기에 더해, **수신자가 스크롤할 때도 유령 블록이 따라 움직여야** 한다.
`timetableScrollTop` ref를 만들고 `@scroll`에서 갱신하도록 해서,
수신자 스크롤이 바뀌면 `collabGhostTop`이 반응형으로 재계산되게 했다.

### 어려움 3 — 투명도 균형

Chrome이 그리는 기본 drag ghost는 약 50% 투명도다.
내가 그린 협업자 유령 블록을 opacity 1로 두니, 내 화면의 흐릿한 드래그와 남의 화면의 선명한 유령이 따로 놀았다.

→ 협업자 유령 블록 opacity를 0.5로 맞춰 Chrome 기본 ghost와 시각적으로 동일하게 했고,
반대로 `setDragImage` clone은 opacity 1로 만들어 **내가 드래그하는 블록 자체는 선명**하게 보이도록 했다.
(clone에 `getComputedStyle`로 배경·테두리색을 직접 복사해야 했다. cssText만으로는 CSS 변수 기반 색이 복제 시점에 풀리지 않았다.)

## 4. 협업자 프로필 이미지 + N+1 회피

커서·아바타에 닉네임 첫 글자 대신 실제 프로필 사진을 띄웠다.

- **N+1 회피**: 협업자가 N명이면 이미지 URL을 N번 조회하게 된다.
  `TripCollaboratorMapper.xml`에서 `attach` 테이블을 JOIN해 **단일 쿼리**로 `profileImageUrl`까지 한 번에 가져오도록 했다.
- **검색 NPE**: 회원 검색 결과를 `Map.of`로 만들고 있었는데, 프로필 이미지가 없는 회원은 URL이 null이라
  `Map.of`에서 NPE가 났다. `HashMap`으로 교체했다. (2번의 broadcast 이슈와 같은 `Map.of` null 함정)
- **race condition**: 이동 시간 재계산 전에 broadcast가 나가면 협업자가 옛 데이터를 받는 순간이 있었다.
  `broadcast` 호출을 `recalculate` 이후로 옮겨 순서를 보장했다.

## 5. 여행 작업실(`/plan`) 구조로 이식

오후에 팀원이 `feature/ui-redesign`을 `master`에 머지했다. 핵심 변화:

- 기존 `ScheduleView.vue`(약 2,200줄 모놀리식)가 **5줄짜리 thin wrapper**로 축소되고,
  실제 일정 보드 로직이 신규 **`ScheduleBoard.vue`** 공용 컴포넌트로 분리됐다.
- 신규 `PlanView.vue`(여행 작업실, `/plan/:tripId`)가 등장. 탐색(explore)↔정리(organize) 모드를 토글하며,
  정리 탭에서 `<ScheduleBoard embedded :trip-id="..." />` 형태로 같은 보드를 재사용한다.

문제는 내 협업 코드가 전부 구버전 `ScheduleView.vue`에 박혀 있었다는 점이다.

### 이식 전략 — rebase 후 컴포넌트로 옮기기

`feature/collab`을 갱신된 `master` 위로 rebase했다. `ScheduleView.vue`는 양쪽이 통째로 달라
충돌 해결보다 **master 버전(thin wrapper)을 그대로 채택**하는 게 깔끔했다.
그 뒤 협업 코드 일체를 `ScheduleBoard.vue`로 옮겼다.

DOM 의존성이 다행히 그대로였다 — `wrapperEl` ref와 `.day-col` 클래스가 신규 컴포넌트에도 동일하게 존재해서,
스크롤·좌표 계산 헬퍼(3번에서 고생한 부분)를 거의 손대지 않고 재사용할 수 있었다.

### 고민했던 포인트 — standalone과 embedded 두 진입 경로

`ScheduleBoard`는 두 가지 방식으로 일정을 받는다.
- **standalone(`/schedule`)**: 자체 드롭다운으로 `activeTripId`를 고른다 → `watch(activeTripId)`에서 WebSocket 연결.
- **embedded(`/plan` 정리탭)**: 부모가 `tripId` prop으로 주입한다 → `watch(() => props.tripId)`에서 연결.

두 경로 모두에서 이전 일정 연결을 끊고(`collab.disconnect()`) 새 일정에 연결(`connectCollab`)하도록 분기를 맞췄다.
`onMounted`도 embedded는 prop으로 바로 로드, standalone은 `list()`+`listCollaborating()` 병렬 로드로 갈라뒀다.
하나의 컴포넌트가 두 생애주기를 동시에 지원해야 해서, 연결/해제 타이밍이 겹치지 않도록 신경 썼다.

## 회고

오늘의 80%는 "기능을 만드는 것"이 아니라 **"좌표를 맞추는 것"**이었다.
실시간 협업의 핵심 난이도는 메시지를 주고받는 인프라가 아니라,
**서로 다른 뷰포트·스크롤·드래그 상태를 가진 두 화면이 같은 한 지점을 가리키게 만드는 일**에 있다는 걸 체감했다.

`grabOffset`, `cursorRelY`, `timetableScrollTop` — 이 세 보정값이 하나라도 빠지면
유령 블록이 미묘하게 어긋나고, 그 어긋남은 사용자에게 "버그"로 보인다.
좌표계의 기준점을 `wrapperEl` 하나로 통일한 게 결국 해법이었다.
