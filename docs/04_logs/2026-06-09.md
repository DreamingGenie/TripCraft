# 작업 일지 — 2026-06-09

## 완료

### 커뮤니티 게시글 상세 페이지 URL 라우팅 분리

---

#### 문제

커뮤니티 페이지에서 게시글을 열어도 URL이 `/community`로 고정되어 있었다.  
이 때문에 브라우저 **뒤로가기**를 누르면 목록이 아닌 이전에 방문한 완전히 다른 페이지로 이동했다.

추가로 새로고침 시 선택한 게시글이 사라지고 목록으로 초기화되었으며,  
특정 게시글 URL을 공유하거나 북마크하는 것도 불가능한 상태였다.

---

#### 원인 분석

기존 구조는 `CommunityView.vue` 하나가 목록과 상세를 모두 담당하며,  
`selectedPost`라는 **JS 내부 상태 변수**로 화면을 전환하는 방식이었다.

```
/community  →  CommunityView.vue
               ├── selectedPost === null  →  목록 화면
               └── selectedPost !== null  →  상세 화면
```

화면은 바뀌어도 URL이 바뀌지 않으므로 브라우저 히스토리 스택에 아무것도 쌓이지 않았다.  
뒤로가기는 히스토리 스택을 역방향으로 소비하는 동작이기 때문에, community 내부 전환은 무시되고  
community 이전에 방문한 페이지로 이동해버렸다.

---

#### 고민한 것들

**`router.back()` vs `router.push('/community')`**

상세에서 목록으로 돌아갈 때 `router.back()`을 쓰면 자연스러워 보이지만,  
사용자가 외부 링크에서 바로 상세 페이지로 진입한 경우 뒤로가기 스택에 `/community`가 없어서  
앱 밖으로 나가버리는 문제가 생긴다.  
`router.push('/community')`로 명시적으로 목록을 지정하는 게 더 안전하다.

**숫자 외 경로 처리**

`/community/:id`에 숫자가 아닌 값(예: `/community/abc`)이 들어오면  
컴포넌트가 마운트된 뒤 API 호출 후 에러를 처리해야 해 불필요한 요청이 발생한다.  
Vue Router의 파라미터 패턴 기능(`\\d+`)을 활용하면 라우터 단에서 즉시 차단할 수 있다.

**보안: 프론트에서 막으면 충분한가?**

충분하지 않다. 프론트 라우팅은 화면 진입을 막는 역할일 뿐이고,  
실제 권한 검증은 서버가 매번 수행해야 한다.  
삭제된 글(`404`) / 비공개 글(`403`) 접근 시도는 모두 서버 응답으로 판단하고  
프론트는 그 결과를 화면에 표시하는 역할만 담당한다.

---

#### 해결 방법

뷰를 목록과 상세로 분리하고, 각각에 독립적인 URL을 부여했다.

```
변경 전
/community  →  CommunityView.vue (목록 + 상세 모두)

변경 후
/community        →  CommunityView.vue     (목록 전용)
/community/:id    →  CommunityPostView.vue (상세 전용)
```

**라우터 경로에 정규식 제약 추가**

```js
{ path: '/community/:id(\\d+)', component: () => import('@/views/CommunityPostView.vue') }
```

`\\d+`로 숫자 ID만 허용. `/community/abc` 등은 라우터 단에서 NotFound로 자동 처리된다.

**상세 뷰에서 접근 에러 처리**

```
/community/999  →  GET /api/posts/999
                →  서버 404  →  "존재하지 않거나 삭제된 게시글입니다."
                →  서버 403  →  "접근 권한이 없는 게시글입니다."
/community/-1   →  isNaN / postId <= 0 검증  →  에러 화면 즉시 표시
```

**공통 유틸 추출**

`formatDate`, `formatTripDate` 함수가 두 뷰에 중복될 상황이므로  
`utils/format.js`로 추출해 양쪽에서 import하도록 했다.

---

#### 수정 파일

| 파일 | 변경 내용 |
|------|----------|
| `router/index.js` | `/community/:id(\\d+)` 라우트 추가 |
| `CommunityView.vue` | 목록 전용으로 축소, 글 클릭 시 `router.push('/community/:id')` |
| `CommunityPostView.vue` | 신규 — `/community/:id` 상세 뷰, 에러 상태 처리 포함 |
| `utils/format.js` | 신규 — `formatDate`, `formatTripDate` 공통 유틸 |

#### 결과

- 뒤로가기가 자연스럽게 목록으로 돌아옴
- 새로고침해도 현재 게시글 유지
- 특정 글 URL 공유·북마크 가능
- 삭제된 글·잘못된 URL 접근 시 에러 화면 표시
- 숫자가 아닌 경로는 라우터 단에서 차단

### 게시글 이미지 첨부 기능 추가 (Tiptap 리치 에디터 도입)

---

#### 구현 의도

커뮤니티 게시글이 텍스트만 지원해 여행 사진·지도 캡처 등을 첨부할 수 없었다.
여행 공유 서비스의 특성상 이미지가 포함된 글이 훨씬 정보 전달력이 높기 때문에,
글쓰기 에디터를 **리치 텍스트 에디터**로 교체하고 이미지 업로드 기능을 함께 도입했다.

---

#### 라이브러리 선택 — Tiptap

에디터 라이브러리 후보를 검색 후 비교했다.

| 라이브러리 | Vue 3 지원 | 라이선스 | 커스터마이징 | 선택 이유 |
|------------|-----------|---------|------------|----------|
| **Tiptap** | 공식 패키지 `@tiptap/vue-3` | MIT | 매우 높음 | ✅ 선택 |
| Quill (VueQuill) | 비공식 래퍼 | BSD | 중간 | |
| TinyMCE | 공식 지원 | 무료 플랜 제한 | 낮음 | |
| CKEditor 5 | 공식 지원 | GPL (상업 유료) | 중간 | |

Tiptap을 선택한 이유:
- Vue 3 공식 패키지가 있어 통합이 간단함
- MIT 라이선스로 제약 없음
- ProseMirror 기반의 확장 시스템으로 이미지·볼드·목록 등을 확장 단위로 조합 가능
- `@tiptap/extension-image`로 이미지 삽입을 공식 지원

DOMPurify를 함께 도입한 이유:
- Tiptap이 출력하는 HTML을 `v-html`로 렌더링할 때 XSS 공격 가능성이 생김
- `<script>`, `onclick` 등 위험 요소를 서버·클라이언트 양쪽에서 제거해야 안전
- DOMPurify는 브라우저 DOM 파서를 활용해 신뢰할 수 있는 태그만 허용하는 표준적 방법

#### 구현 내용

**백엔드**

| 파일 | 역할 |
|------|------|
| `ImageController.java` | `POST /api/images/upload` — 타입·크기 검증 후 서버 저장, URL 반환 |
| `WebMvcConfig.java` | `/uploads/**` 경로로 업로드된 파일 정적 서빙 |
| `application.yml` | `upload.dir`, `multipart.max-file-size: 10MB` 추가 |
| `SecurityConfig.java` | `GET /uploads/**` 인증 없이 접근 허용 |

이미지 저장 흐름:
```
클라이언트 → POST /api/images/upload (multipart)
  → 타입 검증 (JPEG·PNG·GIF·WebP)
  → 크기 검증 (5 MB 이하)
  → UUID 파일명으로 {upload.dir}/images/ 저장
  → 응답: /uploads/images/{uuid}.jpg
```

**프론트엔드**

| 파일 | 역할 |
|------|------|
| `TiptapEditor.vue` | 에디터 컴포넌트 — Bold·Italic·목록·이미지 업로드 툴바 |
| `CommunityView.vue` | 글쓰기 모달 textarea → TiptapEditor 교체 |
| `CommunityPostView.vue` | `v-html` + DOMPurify로 HTML 렌더링 |
| `community.css` | `detail-body` 내 Tiptap HTML 요소 스타일 추가 |

이미지 삽입 흐름:
```
🖼 이미지 버튼 클릭
  → 파일 선택 (accept: image/*)
  → fetch('/api/images/upload', FormData)   ← http 래퍼는 JSON 전용이라 raw fetch 사용
  → 응답 URL을 에디터 커서 위치에 <img> 삽입
  → 글 등록 시 HTML 채로 content 필드에 저장
```

#### 버그 수정 (구현 후 발견)

**1. 백엔드 구동 실패 — application.yml 중복 키**

`spring:` 최상위 키가 파일 내 두 번 선언되어 SnakeYAML이 `DuplicateKeyException`을 던짐.
`servlet.multipart` 설정을 기존 `spring:` 블록 밖에 별도 블록으로 작성한 게 원인.
기존 `spring:` 블록 안으로 병합해 해결.

**2. 이미지 저장 실패 — 상대 경로 기준점 불일치**

`Files.createDirectories()`는 JVM 실행 디렉토리 기준으로 `./uploads`를 생성했지만,
`MultipartFile.transferTo(File)`은 Tomcat 임시 작업 디렉토리 기준으로 경로를 해석해
`FileNotFoundException` 발생. `toAbsolutePath()`로 먼저 절대 경로를 확정해 해결.

**3. 이미지 렌더링 실패 — Vite 프록시 누락**

개발 서버(5173)에서 `/uploads/images/*` 요청이 Vite를 통해 백엔드(8080)로 전달되지 않아
이미지가 깨져 보임. `vite.config.js`에 `/uploads` 프록시 경로 추가해 해결.

**4. 글쓰기 모달 외부 클릭 시 닫힘**

모달 오버레이에 `@click.self="writeModal = false"`가 걸려 있어
Tiptap 에디터 내부 조작 중 의도치 않게 모달이 닫히는 문제 발생. 해당 이벤트 제거.
