### feature/community ← master 머지 및 충돌 해결

feature/mypage가 master에 머지된 후, feature/community를 master와 머지하는 과정에서 충돌이 발생했다.

**충돌 원인:**
두 브랜치가 각각 독립적으로 `FileStorageService`와 `Attach` 관련 코드를 구현했지만 패키지 구조가 달랐다.

| 항목 | feature/community (내 구현) | master (feature/mypage) |
|------|----------------------------|------------------------|
| FileStorageService | `global.service` | `global.storage` |
| Attach 도메인 | `community.domain` | `global.attach.domain` |
| AttachMapper | `community.mapper` | `global.attach.mapper` |
| Attach.targetId | `Long` (nullable) | `long` (primitive) |
| 파일 삭제 | `delete(filename)` | `delete(hostPath)` |

**해결 방향:** master 패키지 구조를 기준으로 통일. community 브랜치에서 추가한 기능(게시글 삭제 시 이미지 cleanup, 고아 파일 스케줄러)은 master API에 맞게 조정해 통합.

**처리 내용:**
- 중복 파일 제거: `global/service/FileStorageService.java`, `community/domain/Attach.java`, `community/mapper/AttachMapper.java`, `mapper/community/AttachMapper.xml`
- `OrphanImageCleanupScheduler`, `PostServiceImpl`, `ImageController` 패키지 경로 업데이트
- `AttachMapper`에 `findExpiredDrafts`, `deleteExpiredDrafts` 추가 (고아 파일 스케줄러용)

### 팀원 수정 내용 확인 — PostMapper.xml MyBatis 크로스 네임스페이스 include 버그

`PostBookmarkMapper`, `PostLikeMapper`가 `PostMapper.myListSelectBase` 조각을
크로스 네임스페이스 include로 참조할 때 앱 기동 실패했던 건을 팀원이 수정했다.

**원인:**

`myListSelectBase` 내부에서 `<include refid="commentCountSubquery"/>` 를 짧은 이름으로
참조하면, MyBatis는 **include를 호출한 쪽의 네임스페이스**를 기준으로 해석한다.
`PostBookmarkMapper`에서 이 조각을 가져오면 `PostBookmarkMapper.commentCountSubquery`를
찾으려 해서 `not found` 에러가 발생한다.

**수정:** `myListSelectBase` 내부 include를 정규화된 refid로 변경

```xml
<!-- 수정 전 -->
<include refid="commentCountSubquery"/>

<!-- 수정 후 -->
<include refid="com.tripcraft.community.mapper.PostMapper.commentCountSubquery"/>
```

MyBatis 크로스 네임스페이스 재사용 시 **참조되는 조각 내부의 include도 전체 경로로 명시**해야 한다.

### 최종 점검 및 보안·품질 개선

#### B-1. FileStorageService — Content-Type 스푸핑 방어 (보안)

**문제:** `file.getContentType()`은 HTTP 요청 헤더 값 그대로 반환한다. 클라이언트가
`.exe` 파일에 `Content-Type: image/jpeg`를 붙여 전송하면 기존 검증을 통과했다.

**수정:** 파일 앞 12바이트(magic bytes) 검증 추가

```java
// JPEG: FF D8 FF
// PNG:  89 50 4E 47
// GIF:  47 49 46 38
// WebP: 52 49 46 46 ?? ?? ?? ?? 57 45 42 50
```

`FileStorageService.save()` 내부에서 Content-Type 검증 → 파일 크기 검증 → magic bytes 검증 순으로 진행한다. 모든 파일 업로드(프로필 이미지, 게시글 이미지)가 공통으로 이 검증을 통과한다.

#### B-2. MemberController — 프로필 이미지 저장 경로 분리

**문제:** `PROFILE_DIR = "images"` 로 설정돼 있어 프로필 이미지와 게시글 본문 이미지가
동일한 `/uploads/images/` 디렉토리에 혼재됐다.

UUID 파일명으로 충돌은 없지만, 운영 중 파일 목적을 구분할 수 없고 이미지 삭제 정책도 구분이 어렵다.

**수정:** `PROFILE_DIR = "profile"`로 변경. 저장 경로 `→ /uploads/profile/UUID.jpg`

```
변경 전: /uploads/images/UUID.jpg  (프로필 + 게시글 이미지 혼재)
변경 후: /uploads/profile/UUID.jpg (프로필)
         /uploads/images/UUID.jpg  (게시글 이미지)
```

WebMvcConfig가 `/uploads/**` 전체를 서빙하므로 추가 설정 불필요.

> ⚠️ **DB 마이그레이션 주의:** 기존 attach 레코드의 `host_path`가 `/uploads/images/` 기준으로
> 저장돼 있다면 `delete()` 호출 시 경로 불일치로 삭제 실패한다. 운영 DB가 있다면
> `UPDATE attach SET host_path = REPLACE(host_path, '/images/', '/profile/') WHERE target = 'profile'`
> 실행 필요.

#### F-1. MyProfileView — Pinia store 상태 직접 변경

**문제:** 닉네임 변경 성공 후 `auth.user.nickname = nicknameForm.value`로 store를 직접 변경했다.

Pinia는 직접 변경을 허용하지만, 서버 응답 없이 클라이언트 상태만 바꾸는 것은 실제 저장 성공 여부와 무관하게 화면에 반영되는 문제가 있다.

**수정:** `await auth.fetchMe()`로 서버에서 최신 상태 재조회

```js
// 수정 전
auth.user.nickname = nicknameForm.value

// 수정 후
await auth.fetchMe()
```

### B-3. PostServiceImpl.deletePost — 파일 삭제 순서

현재 순서: 파일 삭제 → DB 레코드 삭제 → 게시글 soft delete

실무에서는 파일 삭제를 **트랜잭션 커밋 이후**에 수행하는 것이 더 안전하다.
현재 구조에서는 파일이 삭제됐는데 DB 롤백이 발생하면 attach 레코드는 남아 있지만 실제 파일은 없는 불일치 상태가 된다.

개선 방향: Spring의 `@TransactionalEventListener(phase = AFTER_COMMIT)`로 이벤트 발행 후 파일 삭제.
현재 프로젝트 규모와 트랜잭션 실패 빈도를 감안해 미적용.

### F-2. http.js requestForm — 401 retry 시 FormData 재사용

현재 코드는 401 응답 시 토큰을 재발급받고 동일한 `opts` 객체(FormData 포함)로 재요청한다.

`FormData`는 일반적으로 재사용 가능하지만, 브라우저에 따라 첫 번째 `fetch` 호출 시
내부적으로 스트림으로 변환 후 소비되는 경우 두 번째 전송에서 빈 body가 전달될 수 있다.

개선 방향: FormData 요청은 retry 없이 401 즉시 throw하거나, FormData를 클로저로 재생성하도록 변경.
현재는 대부분 브라우저에서 정상 동작하나 엣지 케이스로 기록.
